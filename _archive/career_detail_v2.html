요금제 구조 개편 · 결제 시스템 전환

2026.01 ~ 03
Why

요금제 체크 로직이 13개 페이지에 분산되어 있었다. 코드량은 크지 않았지만 관리 포인트가 흩어져 있어서, AI로 작업할 때 관련 없는 코드까지 컨텍스트에 들어와 원하는 퀄리티가 나오지 않았다. 신규 요금제가 추가되면 레거시와 함께 관리해야 하는데, 이 상태로 가면 AI 활용 효율이 더 떨어지고 이후 개편도 어려워질 거라고 판단해 결제 전환 전에 먼저 구조를 정리했다.

How

기능별 접근 정책을 FeatureMatrix config 한 곳에 선언하고, 라우트 레벨에서 자동 차단하는 구조로 바꿨다. 페이지에서 요금제를 직접 판단하던 코드가 없어졌고, 새 요금제 추가 시 매핑 하나만 수정하면 됐다. 정리 후에는 원하는 동작을 프롬프트에 적으면 원샷으로 작업이 됐다.

결제 PG 전환에서 가장 중요했던 건 이관 작업 중에도 다른 기능 릴리즈가 막히지 않는 것이었다. 당장 가장 빠르게 할 수 있는 방법이 환경변수로 결제 시스템을 분리하는 거였고, 최선인지는 모르겠지만 실제로 전환 기간 동안 신기능 릴리즈에 문제가 없었다. 달리는 차에서 바퀴를 갈아끼운 게 아니라, 차를 두 대 굴린 것에 가깝다.

스타일링 환경 재구축 — MUI/Emotion에서 Tailwind 전환

2025.12 ~ 2026.03
Why

AI로 UI 작업을 했을 때 히트율이 높지 않았다. 스타일 토큰화가 전혀 되어 있지 않았고 공통 컴포넌트도 정리되지 않은 것이 많았다. 이 상태에서 AI가 작업을 진행하면, 일관성 없는 스타일링 구조를 참조하면서 컨텍스트가 오염됐고 결과물 퀄리티가 계속 떨어졌다.

Decision

기반을 MUI/Emotion에서 Tailwind로 전환한 이유는 몇 가지 있었다. MUI/Emotion 위에 스타일링 환경을 구축하면 나중에 되돌리기 어려운 기술부채가 된다고 봤다. MUI 자체도 AI와 궁합이 좋지 않다고 생각했다. Tailwind 기반으로 전환하기로 결정했다.

Result

스타일 토큰화와 공통 컴포넌트 정리를 진행하면서 점진적으로 적용해나갔다. 어느 정도 마무리된 시점에서 피그마 MCP를 사용했을 때 히트율이 체감될 만큼 올라갔다. MUI/Emotion을 전부 걷어내지는 못했지만, 골자는 Tailwind로 옮겼고 AI 레버리지도 크게 올라 만족스러운 결과를 얻었다.

생산성 기여

이 작업이 마무리된 시점 이후로 광고 진단 페이지(AdsDiagnosis)를 4일 만에 완성할 수 있었다. 토큰과 공통 컴포넌트가 정리되면서 페이지를 만들 때 스타일링 판단을 반복하지 않아도 됐기 때문이다.

페이지 스캐폴딩 도구

2026.03 ~
Why

새 페이지를 만들 때 폴더 생성, 기초 컴포넌트 구축, 라우터 등록, 사이드바 메뉴 적용까지 해야 하는데, AI한테 "새 페이지 만들어줘"라고 하면 다른 페이지를 참조해서 폴더와 파일은 만들지만, 라우터에 등록하거나 메뉴에 추가해야 한다는 것까지는 놓칠 때가 종종 있었다.

How

명령어 하나로 컨벤션에 맞는 폴더 구조(components, hooks, types, utils)가 생성되고, 옵션에 따라 API 레이어나 검색·목록 영역도 추가되며, 라우터 등록과 사이드바 메뉴 배치까지 자동으로 완료된다. 1초면 기초 작업이 끝나고, AI가 놓치던 부분이 구조적으로 사라졌다.

Context

신규 기능을 빠르게 만들어 시장 반응을 확인해야 하는 상황이어서 빠른 테스트 환경이 필요했다. 이후 온보딩 페이지 스캐폴딩까지 확장했고, Claude skills로 구축해 비개발자 직원에게 인수인계했다.

GTM 트래킹 체계 분리

2025.01
Why

GTM 트래킹을 도입할 때 기존 trackingHelper.ts에 코드를 추가했다. 이미 600줄이 넘는 파일이었는데, Mixpanel 초기화, Hotjar 설정, 해싱 함수 등이 한 파일에 섞여 있는 상태에서 성격이 다른 도메인 로직이 계속 누적되는 구조가 됐다. 특히 AI로 트래킹 코드를 작업할 때 관심사가 섞여 있는 파일이 오히려 방해가 됐다.

How

GTM 통신은 하나의 클라이언트로 격리하고, 도메인별로 커스텀 훅을 분리해서 이벤트 파라미터에 타입을 걸어두는 방식이었다. 기존 코드를 한꺼번에 옮기는 대신, 새로 추가하는 트래킹만 이 구조로 작성하고 기존 코드는 여유가 될 때 이관하는 전략을 택했다.

Result

실제로 효과를 체감한 건 GA4 이벤트 15개를 추가할 때였다. 분리된 훅 파일에는 동일한 형태의 함수만 있으니까, AI한테 "새 이벤트 하나 추가해"라고 했을 때 틀릴 수가 거의 없는 구조가 됐다. 이 경험이 이후 스타일링 환경이나 요금제 구조를 정비할 때도 같은 기준으로 판단하게 된 출발점이었다.

복잡한 분기 검증용 결제 · 연동 데이터셋 구축

2026.01 ~ 03
Why

Result

외부 상태를 맞추기 위해 반복하던 수동 작업이 사라졌고, 퍼블리싱 검토, QA, 회귀 테스트를 훨씬 빠르게 돌릴 수 있게 됐다.

Context

이 작업은 스타일링 환경 재구축(Tailwind 전환), 요금제 구조 개편과 같은 시기에 병행했다. 환경이 정리되면서 신규 페이지를 만드는 속도가 이전과 체감될 만큼 달랐고, 진단 페이지(AdsDiagnosis)는 상태 분류·모달·GA4 연동까지 포함해 4일, 23개 커밋으로 완성했다.

초기 로딩 성능 개선

2024.11
Why

평소 서비스에 진입하거나 테스트할 때 초기 로딩이 느리다고 느끼고 있었는데, Hotjar 녹화 영상에서 사용자가 흰 화면을 버그로 인식하고 새로고침을 반복하는 장면까지 확인됐다. 서비스에 lazy loading이 하나도 적용되어 있지 않아 메인 번들이 10MB를 넘었다.

Result

React.lazy와 Suspense 기반 코드 스플리팅을 전체 라우트에 적용해 메인 번들을 10MB에서 2MB로 줄였고, 로딩 중에도 fallback UI로 로딩 상태를 표시하도록 변경했다. 이후 흰 화면 버그 리포트가 사라졌다.
