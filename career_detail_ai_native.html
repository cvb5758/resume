FE 병목을 줄이기 위해 구조를 바꾼 작업

기능을 빠르게 만드는 것만으로는 제품 개발 속도를 바꾸기 어렵다고 생각했습니다. 실제로 병목은 화면 구현 자체보다 페이지마다 흩어진 조건 분기, 반복되는 스타일링, 복잡한 상태 검증처럼 개발과 QA 속도를 늦추는 구조에 더 많이 있었습니다. 그래서 기능 개발과 병행해 FE 구조를 공통화했고, 그 결과 새 기능을 시작하는 비용, 변경 비용, 검증 비용을 함께 낮추는 방향으로 정리해왔습니다.

대표 사례 — 요금제·권한 규칙을 한 곳에서 관리하는 구조

플랜과 트랙에 따른 접근 조건이 여러 페이지에 흩어져 있어 신규 요금제 도입이나 정책 변경 때마다 여러 화면을 함께 수정해야 했습니다. 이를 줄이기 위해 요금제·권한 규칙을 한 곳에서 관리하는 구조를 만들고, 페이지 안에 흩어져 있던 조건 분기를 라우트와 정책 중심으로 옮겼습니다. 이후에는 기능별 접근 정책을 한 곳에서 관리할 수 있게 되어 변경 비용이 낮아졌습니다.

대표 사례 — 스타일링 체계 정리

스타일 토큰과 공통 UI가 충분히 정리되지 않은 상태에서는 화면을 만들 때마다 비슷한 스타일링 판단을 반복해야 했고, styled / sx 중심 패턴도 계속 누적되고 있었습니다. 신규 코드와 정리하는 영역에서는 Tailwind와 디자인 토큰을 우선 적용하고, 페이지 래퍼, 로딩, 빈 상태 처리 같은 반복 UI를 공통 컴포넌트로 재구성했습니다. 그 결과 화면별 중복 구현과 스타일링 비용을 줄일 수 있었습니다.

대표 사례 — 검증용 테스트 데이터 환경 정리

검증용 테스트 데이터 환경과 분기 상태 재현 도구를 정비해, 복잡한 결제/연동 상태를 로컬에서 빠르게 재현할 수 있게 했습니다. 광고 진단 영역도 API / component / hook / modal 구조로 분리하고 테스트 타입 설정까지 정리해, 이후 기능 확장과 회귀 검증이 쉬운 형태로 바꿨습니다. 결과적으로 QA와 회귀 확인에 들어가는 비용을 줄이고, 변경 이후 검증 속도를 높일 수 있었습니다.

AI Native 환경을 만든 작업

AI를 단순히 코드 생성 도구로만 쓰기보다, 반복적인 FE 작업 볼륨을 줄여 실제 생산성 레버리지를 만드는 방향으로 활용해왔습니다. 다만 코드베이스 구조와 규칙이 정리되어 있지 않으면 결과 품질이 쉽게 흔들렸기 때문에, 생성 규칙, 폴더 구조, 공통 패턴, 검증 장치를 함께 정비해 AI가 더 적은 비용으로 기능 시작·수정·검증에 기여할 수 있는 환경을 만들었습니다.

대표 사례 — 페이지 기본 구조를 빠르게 만드는 도구

새 페이지를 만들 때마다 폴더 생성, 라우터 등록, 메뉴 연결, 기본 구조 세팅을 반복하고 있었고, 이런 기초 작업은 번거롭지만 빠뜨리기 쉬운 영역이었습니다. 이를 줄이기 위해 페이지 기본 구조를 빠르게 만드는 도구를 구성해, 프로젝트 컨벤션에 맞는 페이지/컴포넌트/훅/API 구조를 빠르게 만들 수 있게 했습니다. 반복적인 초기 작업을 구조로 고정해, 새 기능을 시작하는 비용을 낮추고 AI가 일관된 형태로 작업할 수 있는 기반을 만들었습니다.

대표 사례 — 스타일 기준과 토큰 사용 규칙 점검

AI가 작업할 때 스타일 기준이 흐려지거나 토큰 사용 규칙이 깨지면 결과물이 쉽게 일관성을 잃었습니다. 이를 방지하기 위해 스타일 기준과 토큰 사용 규칙 위반을 확인하는 스크립트를 추가하고, 검증용 테스트 데이터 환경도 dev/stg에서 사용할 수 있게 정리했습니다. 생성 속도만 높이는 것이 아니라, 코드베이스가 쉽게 무너지지 않도록 관리하는 쪽에 더 집중했습니다.

대표 사례 — 트래킹 구조 분리

트래킹 관련 코드가 한 파일에 섞여 있던 시기에는 관심사가 뒤섞여 있어 수정과 확장이 모두 불편했습니다. 새로 추가하는 이벤트부터는 공통 구조와 도메인별 훅으로 분리하고, 이벤트 파라미터에도 타입을 걸어 관리했습니다. 이후에는 같은 형태의 패턴 안에서 작업할 수 있게 되어 반복적인 트래킹 추가와 수정이 훨씬 단순해졌습니다.

가설을 기능으로 옮기고 검증해온 작업

이커머스 SaaS에서 1인 프론트엔드로 일하며, 기획 논의에 참여하고 개발, 출시, 운영까지 신규 기능이 실제 서비스 안에서 동작하도록 만드는 일을 반복해왔습니다. 다만 이 버전에서는 기능을 만들었다는 사실보다, 그 기능이 이후 확장과 검증에서 다시 비용이 커지지 않도록 구조와 흐름을 함께 잡아 생산성 레버리지로 이어지게 한 점에 더 초점을 맞췄습니다.

대표 사례 — 광고 기능 확장

광고 기능을 분석 / 캠페인 / 진단 흐름으로 확장하고, 기획된 AI 채팅과 코칭노트 연결 흐름이 실제 서비스 안에서 동작하도록 구현했습니다. 단일 대시보드를 추가한 것이 아니라 광고 성과를 확인하고, 문제를 해석하고, 다음 액션으로 이어지는 사용 흐름을 실제 화면과 상태로 구체화한 작업이었습니다.

대표 사례 — 수익성 분석 기능

상품별 순이익, 로켓그로스 순이익 계산기, 데이터 조회 기능 등 수익성 분석 기능을 구현했습니다. DataGrid 기반 그룹 분석, 전체화면, 컬럼 접기 등 복잡한 데이터 조회 경험도 함께 개선해, 사용자가 데이터를 더 실제로 읽고 해석하기 쉬운 형태로 다듬었습니다.

대표 사례 — 크롬 익스텐션

소싱 사용자 유입 채널을 검증하기 위해 크롬 익스텐션의 기획 논의에 참여하고 개발을 맡았습니다. 실제 사용 흐름 안에서 문제를 확인하고, 서비스 외부 접점에서 유저를 데려오는 실험을 해본 작업이었습니다. 결과적으로 다운로드는 확보했지만 결제 전환은 낮았고, 이를 통해 우리가 타겟하는 고객층과의 간극도 확인할 수 있었습니다.

대표 사례 — 비즈니스 마스터

대표님이 엑셀로 정리해주신 가설 검증용 대시보드를 바탕으로, 매출 / 광고 / 비용 / 상품 4개 리포트의 기초 화면을 1~2일 안에 구현했습니다. 데이터도 기존 흐름을 그대로 재사용하기보다 Apollo / GraphQL을 연결해 붙였고, 핵심은 엑셀 기반 가설을 빠르게 검증 가능한 UI로 만든 실행 작업이었다는 점이었습니다.

FE 밖으로 확장해 데이터 구조까지 다룬 작업

FE 병목을 줄이려다 보니 자연스럽게 API 응답과 데이터 구조까지 함께 보게 됐습니다. 화면 구현 속도와 정확도를 높이려면, 프론트에서 실제로 필요한 기준에 맞게 응답 구조와 도메인 모델도 함께 정리되어야 한다고 생각했기 때문입니다.

대표 사례 — 광고 기능에 필요한 응답 구조 정리

광고 캠페인 목록과 진단 목록에서 실제로 필요한 데이터를 기준으로 API 응답 구조를 함께 수정했습니다. 단순히 화면에서 우회 처리하는 대신, 프론트와 백엔드가 같은 기준으로 데이터를 다룰 수 있게 맞춘 작업이었습니다.

대표 사례 — 광고 계정 연동에 필요한 데이터 구조와 저장 흐름 정리

광고 계정 연동 기능 개발 과정에서 필요한 저장 구조와 CRUD를 추가하고, 중복 데이터나 활성 상태 레코드 조회 문제도 함께 수정했습니다. 기능이 동작하는 수준을 넘어서 데이터 정합성까지 맞추려 했던 작업이었습니다.

대표 사례 — 운영용 API 기준 정리

운영에 필요한 API 문서와 응답 구조도 함께 정리했습니다. 기능 구현에 필요한 구조를 맞추는 것뿐 아니라, 이후 다른 사람이 같은 맥락으로 이어서 작업할 수 있는 기준도 남기려 했습니다.
