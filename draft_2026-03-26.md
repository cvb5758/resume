## About

AI를 적극적으로 활용하는 프론트엔드 개발자입니다. AI가 좋은 결과를 내려면 코드 환경 자체가 정돈되어 있어야 한다고 생각하고, 실제로 그 환경을 만드는 데 시간을 씁니다. 요금제 로직이 여러 페이지에 흩어져 있으면 먼저 구조를 정리하고, 스타일링 기준이 없으면 토큰 체계부터 잡는 편입니다. 다만 구조 정리 자체가 목적은 아니고, 정리한 뒤에 실제로 AI 레버리지가 올라가는지를 기준으로 판단합니다. 효과가 없으면 인정하고, 있으면 그 방향으로 더 밀고 나갑니다.

## 프로젝트

### 1. 요금제 구조 개편 · 결제 시스템 전환

요금제 체크 로직이 13개 페이지에 분산되어 있었다. 코드량은 크지 않았지만 관리 포인트가 흩어져 있어서, AI로 작업할 때 관련 없는 코드까지 컨텍스트에 들어와 원하는 퀄리티가 나오지 않았다. 신규 요금제가 추가되면 레거시와 함께 관리해야 하는데, 이 상태로 가면 AI 활용 효율이 더 떨어지고 이후 개편도 어려워질 거라고 판단해 결제 전환 전에 먼저 구조를 정리했다.

기능별 접근 정책을 FeatureMatrix config 한 곳에 선언하고, 라우트 레벨에서 자동 차단하는 구조로 바꿨다. 페이지에서 요금제를 직접 판단하던 코드가 없어졌고, 새 요금제 추가 시 매핑 하나만 수정하면 됐다. 정리 후에는 원하는 동작을 프롬프트에 적으면 원샷으로 작업이 됐다.

결제 PG 전환에서 가장 중요했던 건 이관 작업 중에도 다른 기능 릴리즈가 막히지 않는 것이었다. 당장 가장 빠르게 할 수 있는 방법이 환경변수로 결제 시스템을 분리하는 거였고, 최선인지는 모르겠지만 실제로 전환 기간 동안 신기능 릴리즈에 문제가 없었다. 달리는 차에서 바퀴를 갈아끼운 게 아니라, 차를 두 대 굴린 것에 가깝다.

### 2. 복잡한 분기 검증용 결제 · 연동 데이터셋 구축

신규·레거시 요금제가 함께 운영되고 있었고, 같은 티어처럼 보여도 guardrail 차이 때문에 서로 다른 온보딩이나 페이지를 보게 되는 경우가 있었다. 여기에 쇼핑몰·광고 연동 상태까지 함께 엮이면서 화면 분기가 더 복잡해졌는데, 기존에는 결제는 어드민에서 직접 수정해야 했고 쇼핑몰은 실제로 연동하거나 이미 연동된 계정을 찾아야 해서 테스트가 번거롭고 비용이 컸다. 요금제 변경과 연동 상태 관련 작업이 이어지면서 더는 이 병목을 그대로 둘 수 없다고 판단했다.

결제/요금제 상태는 dataset으로, 쇼핑몰·광고 연동 상태는 preset으로 모델링했다. 로컬에서 원하는 조합을 바로 바꿔볼 수 있게 만들어, 실제 어드민 데이터나 외부 연동 상태를 건드리지 않고도 온보딩, 페이지 진입, CTA, 안내 문구 같은 분기 UI를 재현할 수 있게 했다. 구현 자체보다 중요한 건 테스트 입력을 코드로 통제 가능한 형태로 바꾼 것이었다.

외부 상태를 맞추기 위해 반복하던 수동 작업이 사라졌고, 퍼블리싱 검토, QA, 회귀 테스트를 훨씬 빠르게 돌릴 수 있게 됐다. 단순히 mock 데이터를 추가한 것이 아니라, 복잡한 분기를 병목 없이 검증할 수 있는 개발 환경을 만든 셈이었다.

### 3. 숫자 0과 미입력을 구분하지 못하는 문제 추적 · 수정

매입원가에 0을 입력하면 미입력(-)으로 표시된다는 버그 리포트를 받았다. 무료 샘플 등 매입가가 실제로 0원인 상품을 관리할 수 없는 상태였다.

원인을 추적해보니 단일 지점 문제가 아니었다. 프로젝트 공통 유틸 함수가 숫자 0을 "빈 값"으로 판정하고 있었고, 이 함수를 그리드의 모든 숫자 포매터가 사용하고 있어서 0이 입력된 셀이 전부 -로 표시됐다. 거기에 매입원가 계산 로직에서도 JavaScript의 falsy 특성(!0 === true, 0 || 기본값 === 기본값)을 고려하지 않은 패턴이 입력값 변환, 부가세율 초기화, 합산 계산 등 여러 곳에 걸쳐 있었다.

모든 null 체크를 === null || === undefined로 교체하고, 미입력 상태와 0 입력을 구분할 수 있도록 값 변환 로직을 정비했다. 같은 falsy 패턴이 계산 로직 곳곳에도 있어서 순이익·마진율 계산에도 영향이 있을 수 있다고 판단해 함께 수정했지만, 실제로 계산 결과가 어디까지 왜곡되고 있었는지를 정밀하게 검증하지는 못했다.

### 4. 크롬 익스텐션 — 소싱 유저 유입 채널 검증

소싱하는 사용자들을 서비스에 끌어들일 방법을 고민하다가, 경쟁사들이 익스텐션으로 홍보하는 것을 발견했다. 익스텐션은 사용자 브라우저에서 실행되기 때문에 DOM 접근이 자유롭고 봇 탐지 우회 등 이점이 있어서 채택된 방식이라고 판단했다.

직접 소싱을 해보면서 불편한 점을 찾았고, 쿠팡·네이버에서 상품을 클릭하면 익스텐션 마진 계산기에 바로 주입되어 각 플랫폼 수수료가 적용된 예상 마진을 즉시 확인할 수 있게 만들었다. 발굴한 아이템을 서비스에서 저장·관리할 수 있게 연결하면 소싱 유저를 유입시킬 수 있다고 판단해 기획부터 개발까지 독립적으로 진행했다.

최신 기준 1,000+ 다운로드를 기록했지만, 실제로 서비스에 유입되어 결제까지 이어지는 비율은 현저히 낮았다. 우리가 타겟하는 고객층이 아니었다는 결론이 나왔다.

### 5. 스타일링 환경 재구축 — MUI/Emotion에서 Tailwind 전환

AI로 UI 작업을 했을 때 히트율이 높지 않았다. 스타일 토큰화가 전혀 되어 있지 않았고 공통 컴포넌트도 정리되지 않은 것이 많았다. 이 상태에서 AI가 작업을 진행하면, 일관성 없는 스타일링 구조를 참조하면서 컨텍스트가 오염됐고 결과물 퀄리티가 계속 떨어졌다. AI가 완성도 높은 결과를 내놓으려면 코드 환경 자체가 정돈되어 있어야 한다고 판단했고, 스타일링 체계를 먼저 고치기로 했다.

기반을 MUI/Emotion에서 Tailwind로 전환한 이유는 몇 가지 있었다. Next.js 마이그레이션을 고려했을 때 runtime CSS는 비권장이었고, 이 상태에서 MUI/Emotion 위에 스타일링 환경을 구축하면 나중에 되돌리기 어려운 기술부채가 된다고 봤다. MUI 자체도 AI와 궁합이 좋지 않았다. 이미 만들어진 컴포넌트의 스타일을 커스텀해야 하는 구조여서, 원하는 결과를 얻으려면 결국 문서를 직접 찾아봐야 하는 일이 잦았다. 반면 피그마 MCP로 디자인을 가져올 때는 Tailwind 쪽이 훨씬 수월했다.

스타일 토큰화와 공통 컴포넌트 정리를 진행하면서 점진적으로 적용해나갔다. 어느 정도 마무리된 시점에서 피그마 MCP를 사용했을 때 히트율이 체감될 만큼 올라갔다. 이전에는 MCP를 사용해도 레이아웃은 반드시 수작업으로 잡아야 했는데, 이후에는 디테일한 부분만 손보면 되는 수준이었고, 때로는 그마저도 필요 없을 정도의 완성도가 나왔다.

다만 MUI/Emotion을 전부 걷어내지는 못했다. 최대한 작업한 결과가 MUI + Tailwind 혼용 수준이었지만, 골자는 Tailwind로 옮겼고 AI 레버리지도 크게 올라 만족스러운 결과를 얻었다. 최종적으로는 토큰과 공통 컴포넌트를 Pencil이나 Stitch에 넣어 디자이너 공수까지 줄이고 싶었지만, 거기까지는 도달하지 못했다.

### 6. 초기 로딩 성능 개선

평소 서비스에 진입하거나 테스트할 때 초기 로딩이 느리다고 느끼고 있었는데, Hotjar 녹화 영상에서 사용자가 흰 화면을 버그로 인식하고 새로고침을 반복하는 장면까지 확인됐다. 확인해보니 서비스에 lazy loading이 하나도 적용되어 있지 않아 메인 번들이 10MB를 넘었고, fallback UI도 없어서 다운로드가 끝날 때까지 흰 화면이 그대로 노출되고 있었다.

React.lazy와 Suspense 기반 코드 스플리팅을 전체 라우트에 적용해 메인 번들을 10MB에서 2MB로 줄였고, 로딩 중에도 fallback UI로 로딩 상태를 표시하도록 변경했다. 이후 간간이 들어오던 흰 화면 버그 리포트가 사라졌다.

### 7. GTM 트래킹 체계 분리

GTM 트래킹을 도입할 때 기존 trackingHelper.ts에 코드를 추가했다. 이미 600줄이 넘는 파일이었는데, Mixpanel 초기화, Hotjar 설정, 해싱 함수 등이 한 파일에 섞여 있는 상태에서 회원가입·결제·쇼핑몰 연동 같은 GTM 이벤트까지 들어가면서 성격이 다른 도메인 로직이 계속 누적되는 구조가 됐다. 당장 작업이 안 되는 건 아니었지만, 광고 분석 쪽 트래킹이 본격적으로 붙을 예정이었고 그때마다 이 파일이 끝없이 커지는 방향이 맞는지 의문이 들었다.

특히 AI로 트래킹 코드를 작업할 때가 걸렸다. 인간 입장에서는 파일 하나에 다 모여있는 게 스크롤만 하면 되니까 오히려 편할 수도 있는데, AI한테는 관심사가 섞여 있는 파일이 오히려 방해가 됐다. GA4 이벤트만 다루는 구조로 분리하면 AI가 판단할 게 없어지니까 정확도가 올라갈 거라고 생각했다.

GTM 도입 초기에 shared/tracking/ 구조를 따로 설계했다. GTM 통신은 하나의 클라이언트로 격리하고, 도메인별로 커스텀 훅을 분리해서 이벤트 파라미터에 타입을 걸어두는 방식이었다. 기존 코드를 한꺼번에 옮기는 대신, 새로 추가하는 트래킹만 이 구조로 작성하고 기존 코드는 여유가 될 때 이관하는 전략을 택했다.

실제로 효과를 체감한 건 광고 분석 GA4 이벤트 15개를 추가할 때였다. 분리된 훅 파일에는 동일한 형태의 함수만 있으니까, AI한테 "새 이벤트 하나 추가해"라고 했을 때 틀릴 수가 거의 없는 구조가 됐다.

### 8. DataGrid 전체화면 · 컬럼 접기

서비스 특성상 DataGrid에 데이터가 많아서 한눈에 보기 어려웠고, 사용자 해상도가 낮으면 더 불편했다. 대표님이 다른 서비스에서 그리드 전체보기와 컬럼 접기 기능을 보고 괜찮겠다고 하셔서, 바로 AG Grid에서 제공하는 기능을 확인하고 구현을 시작했다.

컬럼 접기는 AG Grid 네이티브 기능에 커스텀 아이콘을 적용하는 방식으로 진행했다. 전체화면은 처음에 MUI Dialog의 fullScreen 옵션을 사용했다. prop 하나로 바로 전체화면이 되니까 가장 빠르게 구현할 수 있는 방법이었다.

근데 막상 붙여보니 Dialog가 Portal 기반이라 기존 DOM 트리 밖에 별도로 렌더링되는 구조였고, 그 안에 AG Grid를 새로 마운트할 수밖에 없었다. 같은 데이터를 보여주는 그리드가 2개가 되어버렸고, 전체화면에서 스크롤하고 필터 걸고 정렬한 게 축소하면 다 날아가는 UX 단절이 생겼다.

원인이 "그리드가 2개"라는 게 명확했으니까 Dialog를 바로 걷어내고, 기존 그리드 컨테이너에 position: fixed를 적용하는 방식으로 바꿨다. 그리드 인스턴스는 하나 그대로 두고 감싸는 박스의 CSS만 토글하는 거라, 전체화면을 켜고 꺼도 스크롤·정렬·필터 상태가 그대로 유지된다.

### 9. 페이지 스캐폴딩 도구

새 페이지를 만들 때 폴더 생성, 기초 컴포넌트 구축, 라우터 등록, 사이드바 메뉴 적용까지 해야 하는데, AI한테 "새 페이지 만들어줘"라고 하면 다른 페이지를 참조해서 폴더와 파일은 만들지만, 라우터에 등록하거나 메뉴에 추가해야 한다는 것까지는 생각하지 못할 때가 종종 있었다. 직접 하기엔 어렵지 않지만 번거로웠다. AI가 기존 코드만 보고 라우터·메뉴 등록 패턴까지 자연스럽게 파악할 수 있도록 코드 구조를 정비하는 방법도 있었지만, 그 방향에 확신이 들지 않았고 얻는 것에 비해 리소스가 크다고 판단했다.

AI와 해결 방법을 의논하다 스캐폴딩이라는 방식을 확인하고 적용했다. 명령어 하나로 컨벤션에 맞는 폴더 구조(components, hooks, types, utils)가 생성되고, 옵션에 따라 API 레이어나 검색·목록 영역도 추가되며, 라우터 등록과 사이드바 메뉴 배치까지 자동으로 완료된다. 1초면 기초 작업이 끝나고, AI가 놓치던 부분이 구조적으로 사라졌다.

서비스가 PMF를 찾지 못한 상황이어서 신규 기능을 빠르게 테스트할 수 있는 환경이 필요했다. 이 도구를 만들면서 다른 단순 반복 작업들도 충분히 스캐폴딩으로 자동화할 수 있다고 생각했다. 현재 온보딩 페이지 스캐폴딩을 추가 작업 중이고, 마무리되면 비개발자나 다른 개발자도 간편하게 사용할 수 있도록 Claude·Codex skills로 구축할 예정이다.

## Skills 압축 버전

### React / State

상태를 먼저 성격별로 나눠 설계합니다. 파생 값은 가능하면 계산으로 처리하고, 서버 상태는 React Query, 국소 UI 상태는 로컬 state나 작은 store로 분리하며, 전역 상태는 필요한 경우에만 신중히 사용합니다.

### TypeScript

TypeScript를 제약을 코드로 옮기는 도구로 사용합니다. 정책과 도메인 규칙은 타입, 유니온, 타입 가드, config 객체로 모델링하고, strict는 기존 코드베이스 상황에 맞춰 점진적으로 강화합니다.

### Styling

신규 코드와 정리하는 영역에서는 Tailwind와 디자인 토큰을 우선합니다. 반복되는 패턴은 공통 컴포넌트로 올리고, 스타일 책임이 여러 레이어에 흩어지지 않도록 관리합니다.

### AI Workflow

AI는 구조가 명확할수록 잘 작동한다고 생각합니다. 상태 경계와 규칙을 먼저 정리한 뒤, AI를 구현, 리팩터링, 테스트 보조, 디자인-코드 전환에 활용합니다.
