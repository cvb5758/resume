결제 시스템 전환 및 안정화

(주)뭉클랩 · 장사왕 · 2026.01 ~ 03

PortOne/Inicis 기반 결제를 Toss Payments로 전환하고, 결제 로직 전체를 재설계해 유지보수성과 안정성을 확보한 프로젝트입니다.

결제 로직 83+ → 10개 모듈 매직 스트링 100+ → 0 PlanTier 23종 enum 13페이지 마이그레이션 온보딩 18단계 → 1단계

What I Did

  • 기존(PortOne/Inicis)·신규(Toss) 결제를 동시 운영할 수 있도록 구조를 분리하고, 환경변수 기반 단계적 오픈으로 전환 리스크를 통제했습니다.
  • 결제 관련 매직 스트링 100+개를 제거하고, PlanTier 23종 enum + 타입 가드 체계로 전환해 런타임 오류 가능성을 제거했습니다.
  • 13개 페이지에 플랜 접근 제어 훅을 마이그레이션하고, FeatureMatrix 패턴으로 기능별 접근 권한을 선언적으로 관리할 수 있게 했습니다.
  • 해지·요금제 변경 플로우를 재설계하고, PO와 조율해 해지 예약·grandfather 할인 등 사용자 혼란을 최소화했습니다.

Challenge & Decision

  • 기존 결제 코드가 83+개 파일에 분산되어 있어, 어디를 고쳐야 하는지 파악하는 것 자체가 첫 번째 과제였습니다.
  • 카드사 심사가 개발 서버에서만 가능했고, 결제 작업 중에도 다른 기능을 계속 릴리즈해야 했기 때문에 브랜치 분리가 아닌 런타임 분기를 선택했습니다. 개발 환경에서는 Toss, 운영에서는 Inicis가 동작하는 구조로, 이후 사용자 결제 데이터 기반 자동 감지로 전환했습니다.
  • PlanTier enum을 도입할 때 기존 문자열과의 호환을 위해 타입 가드를 함께 작성해 점진적 마이그레이션이 가능하도록 했습니다.

GTM 트래킹 체계 통합

(주)뭉클랩 · 장사왕 · 2025.12 ~

Mixpanel·Hotjar·GTM이 한 파일에 혼재되고, 일부 페이지는 자체 트래킹 유틸리티를 별도로 만들어 사용하던 구조를 6개 도메인 훅 기반의 shared 모듈로 통합한 프로젝트입니다.

6개 도메인 훅 430줄 이벤트별 타입 인터페이스 816 additions / 454 deletions

What I Did

  • 도메인 구분 없이 한 파일(trackingHelper.ts)에 혼재되던 트래킹을 6개 도메인 훅(Account, Auth, Service, Ecommerce, AdAnalysis, Calculator)으로 분리했습니다.
  • 이벤트별 타입 인터페이스를 정의해 잘못된 페이로드를 컴파일 타임에 차단하고, dataLayer push 전 이전 파라미터 초기화로 데이터 오염을 방지했습니다.
  • 새 이벤트 추가를 constant → type → hook 패턴으로 단순화하고, 산재된 로컬 트래킹 유틸리티(adsDiagnosisAnalytics.ts 등)를 shared 모듈로 통합했습니다.
  • GTM/GA4를 Mixpanel/Hotjar와 완전 분리해, 기존 Mixpanel 트래킹에 영향 없이 GA4 체계를 독립적으로 구축했습니다.

Challenge & Decision

  • 기존 trackingHelper.ts에 Mixpanel·Hotjar·GTM이 혼재되어 있었고, 일부 페이지는 자체 트래킹 유틸리티를 162줄짜리로 별도 작성해 사용하고 있었습니다.
  • GTM/GA4만 도메인 훅으로 분리하고 Mixpanel/Hotjar는 기존 헬퍼에 그대로 두는 방식으로, 전면 교체 없이 점진적으로 전환했습니다.
  • 도메인별 훅으로 분리해 컴포넌트는 "무엇을 추적할지"만 선언하고, "어떻게 보낼지"는 훅이 담당하게 해 관심사를 분리했습니다.

디자인 시스템 구축 · Tailwind 전환

(주)뭉클랩 · 장사왕 · 2025.07 ~ 2026.03

디자인 시스템을 0에서 설계해 단일 소스에서 CSS·Tailwind·MUI 테마를 자동 생성하는 파이프라인을 구축하고, 레거시 565개 파일을 전환한 뒤 CI 게이트로 회귀를 차단했습니다. 결과적으로 하루 이상 걸리던 UI 작업을 1시간 이내로 단축했습니다.

UI 작업 시간 하루 → 1시간 이내 대상 565개 파일 CSS 커스텀 프로퍼티 202개 15 토큰 카테고리 · ~215 토큰

What I Did

  • 디자인 시스템을 0에서 설계 — 단일 source.js에서 CSS 커스텀 프로퍼티 202개 + Tailwind config + MUI 테마를 동시 생성하는 파이프라인을 구축했습니다.
  • 토큰 기반 래퍼 컴포넌트(PageWrapper, ListContainer, Modal 등)를 만들어 화면 구성 시 스타일링 의사결정 비용을 제거했습니다.
  • AI 협업 워크플로우(기준 문서화 → 구조 정리 → AI 전환 → 수동 검수 → 이슈 환류)로 565개 파일 대규모 전환을 실행했습니다.
  • 스타일 경계 검사 CI 게이트(check-style-system-boundary.mjs)를 적용해 전환 완료 컴포넌트에 MUI sx/styled() 재유입 시 배포를 자동 차단했습니다.

Challenge & Decision

  • 기존에는 스타일링 기준이 없어 같은 UI를 만들 때마다 색상·간격·레이아웃을 매번 판단해야 했고, 이것이 하루 이상 소요의 주된 원인이었습니다.
  • 토큰화 자체보다 "단일 소스 → 3개 포맷 자동 생성" 파이프라인이 핵심이었습니다. 토큰을 수정하면 CSS·Tailwind·MUI가 동시에 반영되어 일관성이 자동으로 보장됩니다.
  • Tailwind v4 도입이 목표였지만 MUI 5가 호환되지 않아, MUI 5→7 업그레이드를 선행한 뒤 Tailwind 전환을 진행하는 2단계 전략을 선택했습니다.
  • 전환 후 회귀를 막기 위해 CI 게이트를 도입했고, "전환했는데 다시 돌아가는" 문제를 구조적으로 차단했습니다.

46개 라우트 lazy loading 4-chunk 번들 분리 6개 멀티앱 빌드 타겟

매입원가 관리 — 입력률 개선을 위한 반복 UX 개선

(주)뭉클랩 · 장사왕 · 2025.11 ~ 12

매입원가 입력률은 서비스에서 가장 중요한 지표였습니다. 매입가가 입력되어야 순이익이 계산되고, 순이익을 보는 셀러만이 서비스에 의존하게 됩니다. 3주간 10회의 반복 개선으로 입력 마찰을 줄이고, 입력 즉시 결과를 보여주며, 서비스 전체에서 매입원가 페이지로의 진입 경로를 연결했습니다.

v1.4.0 ~ v1.4.7 (3주간 10회 릴리즈) 진입 경로 4개 페이지 연결 DataComparisonUtil 신규 설계

What I Did

  • 매입가를 입력해도 결과를 바로 알 수 없던 구조에서, 예상순이익·예상마진율 컬럼을 그리드에 추가해 입력 즉시 수익성을 확인할 수 있게 했습니다. 컬럼 순서도 매입가 → 결과가 바로 옆에 오도록 재배치했습니다.
  • 그리드와 계산기 위젯이 별도로 동작하던 구조를 양방향 동기화로 연결하고, 원본 데이터와의 비교 유틸리티(DataComparisonUtil)를 설계해 실제 변경분만 저장 요청하도록 개선했습니다.
  • 대시보드·주문목록·상품별 순이익·주문 상세 분석 4개 페이지의 "매입가 입력" 버튼에 현재 검색 조건(날짜, 쇼핑몰, 상품ID)을 파라미터로 전달해, 매입원가 페이지 진입 시 맥락이 유지되도록 했습니다.
  • !vatRate 조건이 0을 falsy로 처리해 부가세율 0%를 입력할 수 없던 문제를 === null || === undefined로 수정해, 면세·영세율 상품의 정확한 데이터 입력을 가능하게 했습니다.

Challenge & Decision

  • 매입원가 페이지는 "입력해야 하는 페이지"인데, 입력의 동기가 약했습니다. 매입가를 넣어도 결과를 보려면 다른 페이지로 이동해야 했고, 다른 페이지에서 넘어오면 검색 조건이 초기화되어 다시 설정해야 했습니다. 이 마찰이 입력률 저하의 원인이라고 판단했습니다.
  • "입력하면 즉시 결과가 보인다"와 "어디서든 맥락을 유지하며 진입할 수 있다"를 두 축으로, 예상순이익 컬럼 추가 → 크로스 페이지 파라미터 전달 → 변경분 감지 → 0/null 엣지케이스 순서로 개선했습니다.
  • 기본 필터를 "판매 상품만"에서 "미판매 포함 전체"로 변경한 것은 트레이드오프였습니다. 데이터가 많아져 로딩이 느려질 수 있지만, 매입가가 아직 입력되지 않은 상품을 먼저 보여주는 것이 입력률에 더 중요하다고 판단했습니다.
