디자인 시스템 구축 · Tailwind 전환

(주)뭉클랩 · 장사왕 · 2025.07 ~ 2026.03

화면마다 제각각이던 스타일 기준을 통일하기 위해 디자인 시스템을 0에서 설계하고, 단일 디자인 토큰 소스에서 CSS·Tailwind·MUI 테마를 자동 생성하는 파이프라인을 구축했습니다. 레거시 565개 파일을 전환한 뒤 CI 게이트로 회귀를 차단했습니다. 결과적으로 하루 이상 걸리던 UI 작업을 1시간 이내로 단축했습니다.

UI 작업 시간 하루 → 1시간 이내 대상 565개 파일 3개 스타일 포맷 동시 생성 회귀 차단 CI 게이트

What I Did

  • 디자인 시스템을 0에서 설계하고, 단일 디자인 토큰 소스에서 CSS 변수·Tailwind 설정·MUI 테마를 동시에 생성하는 파이프라인을 구축했습니다.
  • 토큰 기반 래퍼 컴포넌트(PageWrapper, ListContainer, Modal 등)를 만들어 화면 구성 시 스타일링 의사결정 비용을 제거했습니다.
  • AI 협업 워크플로우(기준 문서화 → 구조 정리 → AI 전환 → 수동 검수 → 이슈 환류)로 565개 파일 대규모 전환을 실행했습니다.
  • 스타일 경계 검사 CI를 적용해 전환 완료 컴포넌트에 기존 스타일링 방식이 다시 들어오면 배포를 자동 차단했습니다.

Challenge & Decision

  • 기존에는 스타일링 기준이 없어 같은 UI를 만들 때마다 색상·간격·레이아웃을 매번 판단해야 했고, 이것이 하루 이상 소요의 주된 원인이었습니다.
  • 토큰화 자체보다 "단일 소스 → 3개 포맷 자동 생성" 파이프라인이 핵심이었습니다. 토큰을 수정하면 CSS·Tailwind·MUI가 동시에 반영되어 일관성이 자동으로 보장됩니다.
  • Tailwind v4 도입이 목표였지만 MUI 5가 호환되지 않아, MUI 5→7 업그레이드를 선행한 뒤 Tailwind 전환을 진행하는 2단계 전략을 선택했습니다.
  • 전환 후 회귀를 막기 위해 CI 게이트를 도입했고, "전환했는데 다시 돌아가는" 문제를 구조적으로 차단했습니다.

결제 시스템 전환 및 안정화

(주)뭉클랩 · 장사왕 · 2026.01 ~ 03

기존 PG 기반 결제를 새 PG인 Toss Payments로 전환하면서, 구독 정책의 모호함과 시스템 간 불일치를 사용자 시나리오 기준으로 정리한 프로젝트입니다.

결제 로직 83+ → 10개 모듈 매직 스트링 100+ → 0 요금제 접근 정책 타입화 13페이지 마이그레이션 구/신 결제 동시 운영

What I Did

  • 구/신 결제를 동시 운영할 수 있도록 런타임 분기 구조를 설계하고, 개발/운영 환경을 분리해 점진적으로 전환했습니다.
  • 해지 예약·요금제 변경·기존 할인 유지 등 구독 플로우를 구현하면서, 정책이 모호하거나 사용자 혼란을 줄 수 있는 케이스를 발견해 PO·백엔드와 함께 정리하고 개선했습니다.
  • 백엔드 저장 방식과 API 제약을 함께 확인해 누락된 케이스를 드러내고, 처리 기준과 사용자 안내 방식을 구체화했습니다.
  • 결제/요금제 조건을 문자열 비교 대신 타입 기반 정책으로 정리하고, 13개 페이지 접근 제어를 점진적으로 마이그레이션했습니다.

Challenge & Decision

  • 기존 결제 코드가 83+개 파일에 분산되어 있어, 어디를 고쳐야 하는지 파악하는 것 자체가 첫 번째 과제였습니다.
  • 카드사 심사가 개발 서버에서만 가능했고, 결제 작업 중에도 다른 기능을 계속 릴리즈해야 했기 때문에 브랜치 분리가 아닌 런타임 분기를 선택했습니다. 개발 환경에서는 새 PG, 운영에서는 기존 PG가 동작하는 구조로 시작한 뒤, 이후 사용자 결제 데이터 기반 자동 감지로 전환했습니다.
  • 정책이 문서보다 사람마다 다르게 이해되고 있었기 때문에, 화면부터 만들기보다 사용자 케이스와 저장/API 제약을 먼저 맞춰서 어디서 UX가 깨지는지 드러낸 뒤 정책을 구체화했습니다.
  • 기존 문자열 기반 로직과의 호환 계층을 함께 두어, 운영 중인 화면을 한 번에 뒤집지 않고 점진적으로 옮길 수 있게 설계했습니다.

웹 성능 및 런타임 안정화

(주)뭉클랩 · 장사왕 · 2024.11 ~

대시보드 첫 진입에 2.5~3.5초가 걸리던 문제를 줄이기 위해, 첫 화면에서 필요하지 않은 무거운 편집기·그리드·SDK를 공통 진입 경로에서 단계적으로 분리한 프로젝트입니다. 1차로 메인 JS를 10.4MB에서 2.4MB로 줄였고, 후속 최적화로 현재 메인 엔트리 청크를 2.53MB에서 683KB까지 추가 축소했습니다.

36개 페이지 lazy loading 1차: 메인 JS 10.4MB → 2.4MB 후속: 메인 엔트리 청크 2.53MB → 683KB 대시보드 첫 진입 2.5~3.5s → 1.5~2.0s

What I Did

  • 정적으로 한 번에 로드되던 36개 페이지를 페이지 단위 지연 로딩으로 전환하고, 준비된 영역부터 먼저 렌더되도록 구성했습니다.
  • AG Grid 초기화와 전역 설정을 앱 공통 경로에서 분리해, 그리드가 필요한 화면에서만 무거운 런타임 비용이 발생하도록 구조를 변경했습니다.
  • 번들 분리와 대시보드 워터폴 로딩 제거로 초기 진입 경로를 단순화했고, 이후 편집기·결제 SDK·추적 SDK까지 사용 시점 로딩으로 고도화했습니다.
  • 1차 최적화에서 메인 JS를 10.4MB에서 2.4MB로 줄였고, 후속 최적화에서 현재 메인 엔트리 청크를 2.53MB에서 683KB 수준까지 추가 축소했습니다.

Challenge & Decision

  • 초기 공통 경로에 AG Grid, Toast UI, tui-grid, 각종 SDK가 함께 얹혀 있어 실제로 사용하지 않는 기능의 비용도 첫 화면에서 모두 지불하는 구조였습니다.
  • 무조건 잘게 쪼개기보다 라우트 단위 분리와 초기화 분리를 먼저 적용해 공통 진입 경로를 줄이고, 사용 빈도가 낮은 기능부터 순차적으로 공통 경로에서 제거하는 단계적 전략을 선택했습니다.
  • 단순 번들 크기 축소보다 실제 체감 대기 시간 개선에 집중해, 대시보드에서 풀페이지 로딩을 부분 렌더링과 병렬화 구조로 전환했습니다.

신규 페이지 스캐폴더 구축 · 초기 셋업 자동화

(주)뭉클랩 · 장사왕 · 2026.03 ~

신규 기능 화면을 만들 때마다 기본 파일/폴더 생성, 라우트·메뉴 연결, 검색/목록형 화면 골격을 반복 작성하던 흐름을 CLI 스캐폴더로 통합한 내부 DX 프로젝트입니다.

8~10개 파일 자동 생성 6~8개 디렉터리 자동 셋업 98~106줄 보일러플레이트 자동화 라우트 · 메뉴 자동 등록 검색/목록형 화면 템플릿

What I Did

  • `npm run scaffold:page -- <FeatureName> [--with-api] [--with-mobile]` 기반으로 신규 화면 기본 구조를 생성하는 CLI를 구축했습니다.
  • 기본 페이지 구조와 optional API/mobile 템플릿을 포함해, 화면 초기 구조를 동일한 패턴으로 시작할 수 있게 만들었습니다.
  • 검색 조건 + 목록 + 페이지네이션이 필요한 화면 템플릿과 라우트·좌측 메뉴 등록 옵션을 추가해, 신규 화면 시작 흐름을 한 번의 명령으로 통합했습니다.
  • 신규 페이지 생성 시 수동으로 만들던 8~10개 파일, 6~8개 디렉터리, 98~106줄 보일러플레이트를 CLI 스캐폴더로 자동화했습니다.

Challenge & Decision

  • 신규 화면을 만들 때마다 기본 파일/폴더 생성, 라우트 등록, 좌측 메뉴 등록, 검색/목록형 화면 구성을 각각 수동으로 처리해야 했고, 이 과정에서 구조 편차가 쉽게 발생했습니다.
  • 단순 폴더 생성기가 아니라 실제 서비스 관례를 반영한 온보딩 도구가 필요했기 때문에, 검색/목록형 화면 패턴과 라우트·메뉴 등록까지 포함하는 방향으로 확장했습니다.
  • 전체 기능을 완성하는 도구가 아니라 "기능 구현에 바로 진입할 수 있는 초기 셋업 자동화"에 범위를 제한해, 과한 추상화 없이 실무에 바로 쓰이는 형태로 유지했습니다.

GTM 트래킹 체계 통합

(주)뭉클랩 · 장사왕 · 2025.12 ~

공용 추적 유틸 하나에 Mixpanel·Hotjar·GTM이 섞여 있어 GA4만 수정해도 영향 범위가 넓던 구조를, GA4 계측 레이어 분리와 타입 강제로 재정리한 프로젝트입니다.

6개 도메인 훅 430줄 이벤트 타입 인터페이스 GA4 레이어 분리 광고 · 회원가입 · 결제 계측 통합

What I Did

  • 공용 추적 유틸에 섞여 있던 GA4 계층을 분리하고, 계정·인증·서비스·이커머스·광고 분석·계산기 도메인별 훅 구조로 재정리했습니다.
  • 이벤트별 타입 인터페이스를 정의해 잘못된 페이로드를 컴파일 타임에 차단하고, dataLayer push 전 이전 파라미터 초기화로 데이터 오염을 방지했습니다.
  • GA4 이벤트 추가·변경이 특정 도메인 훅 안에서 닫히도록 정리해, 광고·회원가입·결제 계측을 더 빠르게 수정할 수 있는 구조로 바꿨습니다.
  • 페이지별로 따로 들고 있던 중복 계측 코드도 공용 모듈로 흡수해 GA4 설정의 유연성을 높였습니다.

Challenge & Decision

  • 기존 공용 추적 코드에 Mixpanel·Hotjar·GTM이 혼재되어 있었고, 일부 페이지는 자체 계측 코드를 별도로 들고 있어 변경 범위가 쉽게 커지는 구조였습니다.
  • 전면 교체 대신 GA4만 우선 분리하고 Mixpanel/Hotjar는 기존 헬퍼에 유지하는 점진적 전환을 선택해, 기존 계측을 깨지 않고 구조를 개선했습니다.
  • 목표는 모든 트래킹 툴 통합이 아니라 "GA4만 작업할 때도 빠르고 안전하게 수정 가능한 구조"를 만드는 것이었기 때문에, 계측 레이어 분리와 타입 안전성 확보에 집중했습니다.
