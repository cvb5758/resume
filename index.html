GTM 트래킹 체계 분리

2025.01
Why

600줄짜리 trackingHelper.ts에 Mixpanel·Hotjar·GTM이 혼재되어, AI 작업 시 관심사가 섞인 파일이 오히려 방해가 됐다.

Result

도메인별 커스텀 훅으로 분리하고 이벤트 파라미터에 타입을 걸어둔 결과, GA4 이벤트 15개 추가 시 AI가 틀릴 수 없는 구조가 됐다.

Why

600줄짜리 trackingHelper.ts에 Mixpanel·Hotjar·GTM이 혼재되어 있었다. 당장 작업이 안 되는 건 아니었지만, 광고 분석 쪽 트래킹이 본격적으로 붙을 예정이었고 이 파일이 끝없이 커지는 방향이 맞는지 의문이 들었다. 특히 AI한테는 관심사가 섞인 파일이 오히려 방해가 됐다.

Result

도메인별 커스텀 훅으로 분리하고 이벤트 파라미터에 타입을 걸어둔 결과, GA4 이벤트 15개 추가 시 AI가 틀릴 수 없는 구조가 됐다. 기존 코드는 한꺼번에 옮기지 않고, 새로 추가하는 트래킹만 이 구조로 작성하는 전략을 택했다.

Why

GTM 트래킹을 도입할 때 기존 trackingHelper.ts에 코드를 추가했다. 이미 600줄이 넘는 파일이었는데, Mixpanel 초기화, Hotjar 설정, 해싱 함수 등이 한 파일에 섞여 있는 상태에서 회원가입·결제·쇼핑몰 연동 같은 GTM 이벤트까지 들어가면서 성격이 다른 도메인 로직이 계속 누적되는 구조가 됐다. 특히 AI로 트래킹 코드를 작업할 때가 걸렸다. 인간 입장에서는 파일 하나에 다 모여있는 게 스크롤만 하면 되니까 오히려 편할 수도 있는데, AI한테는 관심사가 섞여 있는 파일이 오히려 방해가 됐다.

How

GTM 도입 초기에 shared/tracking/ 구조를 따로 설계했다. GTM 통신은 하나의 클라이언트로 격리하고, 도메인별로 커스텀 훅을 분리해서 이벤트 파라미터에 타입을 걸어두는 방식이었다. 기존 코드를 한꺼번에 옮기는 대신, 새로 추가하는 트래킹만 이 구조로 작성하고 기존 코드는 여유가 될 때 이관하는 전략을 택했다.

Result

실제로 효과를 체감한 건 광고 분석 GA4 이벤트 15개를 추가할 때였다. 분리된 훅 파일에는 동일한 형태의 함수만 있으니까, AI한테 "새 이벤트 하나 추가해"라고 했을 때 틀릴 수가 거의 없는 구조가 됐다.

Why

평소 초기 로딩이 느리다고 느끼던 중, Hotjar 녹화에서 사용자가 흰 화면을 버그로 인식하고 새로고침을 반복하는 장면을 확인했다. lazy loading이 전혀 없어 메인 번들이 10MB를 넘었고, fallback UI도 없어서 다운로드가 끝날 때까지 흰 화면이 그대로 노출되고 있었다.

Result

Why

평소 서비스에 진입하거나 테스트할 때 초기 로딩이 느리다고 느끼고 있었는데, Hotjar 녹화 영상에서 사용자가 흰 화면을 버그로 인식하고 새로고침을 반복하는 장면까지 확인됐다. 확인해보니 서비스에 lazy loading이 하나도 적용되어 있지 않아 메인 번들이 10MB를 넘었고, fallback UI도 없어서 다운로드가 끝날 때까지 흰 화면이 그대로 노출되고 있었다.

Result

React.lazy와 Suspense 기반 코드 스플리팅을 전체 라우트에 적용해 메인 번들을 10MB에서 2MB로 줄였고, 로딩 중에도 fallback UI로 로딩 상태를 표시하도록 변경했다. 이후 간간이 들어오던 흰 화면 버그 리포트가 사라졌다.

Why

Why

Result

모든 null 체크를 === null || === undefined로 교체해 모든 필드에서 0과 null이 구분 가능해졌고, 이후 안정적으로 운영되고 있다.

Root Cause

원인을 추적해보니 단일 지점 문제가 아니었다. 프로젝트 공통 유틸 함수가 숫자 0을 "빈 값"으로 판정하고 있었고, 이 함수를 그리드의 모든 숫자 포매터가 사용하고 있어서 0이 입력된 셀이 전부 -로 표시됐다. 거기에 매입원가 계산 로직에서도 JavaScript의 falsy 특성(!0 === true, 0 || 기본값 === 기본값)을 고려하지 않은 패턴이 입력값 변환, 부가세율 초기화, 합산 계산 등 여러 곳에 걸쳐 있었다.

Result

모든 null 체크를 === null || === undefined로 교체하고, 미입력 상태와 0 입력을 구분할 수 있도록 값 변환 로직을 정비했다. 이후 모든 필드에서 0과 null이 정상적으로 구분되며 안정적으로 운영되고 있다.
