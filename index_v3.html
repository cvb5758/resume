동시에 AI를 단순한 코드 생성 도구로만 쓰지 않고, 프로젝트 규칙과 페이지 기본 구조, 검증 환경, 디자인 기준을 먼저 정리해 AI로 작업해도 결과가 흔들리지 않는 환경을 만드는 쪽에 집중해왔습니다. 그리고 화면만 만드는 역할에 머무르지 않고, 화면 밖에서 개발 속도를 늦추는 데이터 구조와 연동 흐름 문제도 함께 보며 개발 속도를 늦추는 병목을 줄여왔습니다.

합류 후에는 주어진 화면만 만드는 사람이 아니라, 팀의 가설을 빠르게 검증하고 PMF를 찾는 과정에 기여하는 프론트엔드로 일할 수 있습니다. 동시에 같은 작업을 반복하지 않도록 구조를 정리해 반복되는 개발 비용까지 함께 줄일 수 있습니다.

일하는 방식.

AI와 일하는 방식

  • AI는 코드 생성 자체보다, 구조와 규칙이 명확할수록 더 안정적으로 작동한다고 생각합니다.
  • 반복 초기 세팅, 조건 분기, 스타일 경계, QA 재현 환경을 먼저 정리한 뒤 활용하고, 구현 속도보다 작은 검증 루프와 작업 기준을 함께 두는 편입니다.

디자이너와 협업하는 방식

  • 의견이 갈릴 때는 말로만 설득하기보다, 케이스별 UI와 인터랙션을 직접 구현해 화면 기준으로 빠르게 합의합니다.
  • 디자인 의도를 존중하되, 사용자 입장에서 혼란이 생길 수 있는 부분은 실제 흐름 안에서 함께 확인하며 조율합니다.

백엔드와 협업하는 방식

  • 이슈가 생기면 화면에서 보이는 현상만 전달하기보다, 원인 지점과 payload·response를 먼저 정리해 공유합니다.
  • 프론트에서 필요한 데이터가 있다면 어떤 UI를 위해 왜 필요한지 함께 설명하고, 가벼운 수정은 직접 PR로 연결해 전체 기능 흐름이 막히지 않게 일합니다.

의사결정자와 협업하는 방식

  • 추상적인 취향보다 왜 이 방향이 더 적합한지에 집중해 설명합니다.
  • 가능한 경우 사용자 흐름, 검증 가능한 화면, 수치나 운영 맥락을 기준으로 판단 근거를 만들고, 빠르게 실행해서 확인할 수 있는 형태로 연결하려고 합니다.
