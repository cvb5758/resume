## 1. 크롬 익스텐션과 스크래핑

### 왜 크롬 익스텐션은 스크래핑에 자유로운가?

핵심은 **브라우저의 신뢰 수준(Trust Level) 모델** 때문이다.

| 출처 | 신뢰 수준 | 제한 |
|------|----------|------|
| 웹페이지 JS | 낮음 | CORS, 샌드박스, Same-Origin 등 강하게 제한 |
| 크롬 익스텐션 | 중간 | 사용자가 설치 시 권한을 승인 → 제한 완화 |
| 브라우저 자체 | 높음 | 거의 제한 없음 |

### CORS 등의 보안 정책의 목적

- CORS는 **"신뢰할 수 없는 원격 코드로부터 사용자를 보호"** 하기 위한 것
- **"사용자 자신이 선택한 도구를 제한"** 하려는 것이 아님

```
웹페이지 JS = 가게에 들어온 손님 → 카운터 뒤로는 못 감
익스텐션 = 사장이 고용한 직원 → 사장(사용자)이 허가했으니 카운터 뒤도 접근 가능
```

### 익스텐션이 유리한 구체적 이유들

1. **사용자 브라우저 컨텍스트에서 실행** - 로그인 세션, 쿠키 등 실제 사용자의 것
2. **CORS 제한 우회** - manifest.json 권한 선언으로 cross-origin 요청 자유
3. **DOM 직접 접근** - Content Script가 페이지 DOM에 직접 주입
4. **IP 차단 회피** - 각 사용자의 개별 IP에서 요청, 트래픽 분산
5. **봇 탐지 우회** - 실제 Chrome 브라우저 안에서 실행, fingerprint 정상
6. **법적 관점** - 사용자가 자신의 브라우저에서 접근 가능한 데이터를 가공하는 것

## 2. Lazy Loading & Suspense 도입 배경

### 문제 상황

```
기존: 사용자 접속 → 메인 번들(10MB 이상, 모든 페이지 포함) 다운로드 → 렌더링
                     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                     이 구간 동안 흰색 화면 (White Screen)
```

- 기존에 lazy loading이 **하나도** 걸려있지 않았음
- 메인 번들이 **10MB를 넘었음**
- fallback UI가 적용되어 있지 않아 번들 다운로드까지 **흰 화면**
- **Hotjar 세션 녹화**에서 사용자가 흰 화면을 버그로 인식하고 **새로고침을 반복**하는 장면 발견

### 해결 방향

1. **`React.lazy()`** → 번들을 페이지 단위로 분리 (code splitting) → 메인 번들 크기 감소
2. **`Suspense fallback`** → 분리된 청크 로드 중 흰 화면 대신 로딩 UI 표시

### 이력서/면접용 표현

> "기존에 lazy loading이 전혀 없어서 메인 번들이 10MB를 넘었고, fallback UI도 없어서 번들 다운로드 중 흰 화면이 노출됐습니다.
> Hotjar 세션 녹화를 분석하다가 실제 사용자가 이걸 버그로 인식하고 새로고침을 반복하는 장면을 발견했고,
> 이걸 계기로 React.lazy와 Suspense 기반 코드 스플리팅을 전체 라우트에 적용했습니다.
> 결과적으로 초기 번들 크기가 대폭 감소했고, 로딩 중에도 Fallback UI가 노출되면서 사용자의 혼란이 해소되었습니다."

**포인트**: "데이터(Hotjar)로 문제를 발견 → 기술적으로 해결 → 측정 가능한 결과" 세 박자

## 3. 구현 아키텍처

### 3-1. 라우트 단위 코드 스플리팅

모든 페이지(약 62개)를 `React.lazy()`로 감싸서 라우트 단위로 청크 분리:

```typescript
// 기존 - 전부 즉시 import
import Dashboard from './pages/Dashboard';
import Orders from './pages/Orders';

// 변경 - 필요할 때만 로드
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Orders = lazy(() => import('./pages/Orders'));
```

### 3-2. 계층적 Suspense 구조

Suspense를 한 곳에만 건 게 아니라 **역할별로 나눠서** 적용:

```
App.tsx
├── Suspense (fallback=GlobalLoading)  ← 라우트 전환 시 로딩 UI
│   └── AppRoutes (모든 lazy 페이지)
├── Suspense (fallback=null)           ← 모달/다이얼로그용
│   └── CommerceGuideDialog, TrialDialog 등
└── Suspense (fallback=null)           ← 개발자 도구
```

### 왜 모달에는 fallback={null}인가?

모달은 **이미 페이지가 렌더링된 상태**에서 열림:

```
페이지 전환 시:
  현재 화면 사라짐 → 새 페이지 청크 로딩 → 새 화면 렌더링
                      ^^^^^^^^^^^^^^^^^^^
                      이 구간에 아무것도 안 보이면 흰 화면 → GlobalLoading 필요

모달 열릴 때:
  현재 페이지 유지 (그대로 보임) → 모달 청크 로딩 → 모달 팝업
                                   ^^^^^^^^^^^^^^^^^^^
                                   페이지는 이미 보이고 있음 → 로딩 UI 불필요
```

- `fallback={null}` = "청크 로드될 때까지 아무것도 추가로 표시하지 마, 기존 화면 그대로 둬"
- 여기에 `GlobalLoading`을 넣으면 오히려 전체 화면이 로딩 스피너로 덮이는 부자연스러운 경험

### 3-3. Vendor 라이브러리 청크 분리 (manualChunks)

Vite의 `manualChunks`로 무거운 라이브러리도 별도 청크로 분리:

```typescript
// vite.config.ts
manualChunks: (id: string) => {
  if (!id.includes('node_modules')) return undefined;

  // React + UI 라이브러리
  if (id.includes('react') || id.includes('@mui/') || id.includes('recharts'))
    return 'vendor-core';

  // AG-Grid (무거운 데이터 테이블)
  if (id.includes('ag-grid-'))
    return 'grid';

  // Firebase 인증
  if (id.includes('firebase/'))
    return 'firebase';
}
```

| 청크 | 포함 내용 |
|------|----------|
| `vendor-core` | React, MUI, Recharts |
| `grid` | AG-Grid (데이터 테이블) |
| `firebase` | Firebase 인증 |
| 페이지별 청크 | 각 라우트마다 별도 파일 |

### 3-4. 핵심 파일 경로

| 파일 | 역할 |
|------|------|
| `src/routes/index.tsx` | 62개의 lazy() 라우트 정의 |
| `src/routes/SuspenseWithLoading.tsx` | GlobalLoading fallback 제공하는 루트 Suspense |
| `src/App.tsx` | 앱 전체 구조, 다중 Suspense 영역 |
| `src/components/Loading/GlobalLoading.tsx` | 로딩 UI (애니메이션) |
| `vite.config.ts` | 번들 최적화 (manualChunks) |

## 4. 면접 답변 정리

### Q. "성능 최적화 작업을 왜 했나요?"

> 기존에 lazy loading이 전혀 없어서 메인 번들이 10MB를 넘었습니다.
> fallback UI도 없어서 번들 다운로드 중 흰 화면이 노출됐고,
> Hotjar 세션 녹화를 분석하다가 실제 사용자가 이걸 버그로 인식하고 새로고침을 반복하는 장면을 발견했습니다.
> 누군가 보고한 버그가 아니라 직접 사용자 행동을 분석하면서 발견한 문제였습니다.

### Q. "어떤 아키텍처/방식으로 구현했나요?"

> 세 가지 레이어로 접근했습니다.
>
> 1. **라우트 단위 lazy loading** - 약 60개 페이지를 모두 React.lazy()로 감싸서 페이지별 청크 분리
> 2. **역할별 Suspense 계층화** - 페이지 전환에는 GlobalLoading fallback, 모달에는 fallback={null}로 각 영역에 맞게 적용
> 3. **vendor 청크 분리** - Vite manualChunks로 React/MUI, AG-Grid, Firebase 등 무거운 라이브러리를 별도 청크로 분리

### Q. "모달에 fallback={null}을 쓴 이유는?"

> 모달이 열릴 때는 이미 페이지가 렌더링된 상태입니다.
> 뒤에 페이지가 보이고 있으니 로딩 UI가 필요 없고,
> 오히려 GlobalLoading을 넣으면 전체 화면이 스피너로 덮이는 부자연스러운 경험이 됩니다.
> fallback={null}은 "기존 화면 그대로 두고, 청크 로드되면 모달만 띄워"라는 의미입니다.

### Q. "결과는 어땠나요?"

> 메인 번들 10MB에서 페이지별 청크 분리로 초기 로드 번들이 대폭 감소했고,
> 로딩 중에도 Fallback UI가 표시되면서 사용자의 새로고침 문제가 해소되었습니다.
> (수치가 있으면 추가: FCP OO초 → OO초, 번들 크기 OO% 감소)
