# 이력서 작성 과정에서 배운 점

## 1. 전체 작업 리스트를 보여줄 때

기술 카테고리별 나열("리팩토링 N건, 버그 수정 N건")은 개발자 시점이라 읽는 사람에게 와닿지 않음. **사용자에게 어떤 가치를 줬는지** 기준으로 묶어야 함.

- "데이터를 더 빠르게 보여주기" (성능)
- "데이터를 더 편하게 다루게 하기" (DataGrid)
- "발견 즉시 행동할 수 있는 동선 설계" (페이지 간 연결)

나열만 하면 "많이 했다"는 인상만 남고 "잘 했다"는 인상은 안 남음. **본문은 2~3개만 깊게**, 전체 리스트는 포트폴리오로 빼기.

## 2. 기술 용어 사용 주의

면접관에게 컨텍스트가 없는 내부 유틸 함수명(`safeNumber`, `toNumberIfExists` 등)은 쓰지 않기. "미입력 상태와 0 입력을 구분할 수 있도록 값 변환 로직을 정비함"처럼 **무엇을 해결했는지**로 표현.

## 3. 과장과 솔직함의 경계

**과장 예시**: "순이익·마진율·부가세 계산 결과 자체가 왜곡될 수 있는 문제였음"
→ 실제로 그걸 정밀하게 확인하지 못했으면 과장.

**솔직한 표현**: "영향이 있을 수 있다고 판단해 함께 수정했지만, 실제로 어디까지 왜곡되고 있었는지를 정밀하게 검증하지는 못했음"

**기준**: 내가 직접 확인한 것만 단정. 추측은 "~할 수 있다고 판단해"로 표현.

## 4. AI 탓으로 돌리지 않기

"AI로 작업했는데 AI가 못 잡았다"고 쓰면 → "본인이 코드를 직접 이해하고 있었는가?" 의문 유발. grep 한번 돌려봤으면 잡을 수 있었던 영역은 AI 탓이 아니라 본인 판단의 문제.

AI 활용 이야기는 해당 버그 항목이 아니라 **About이나 별도 항목에서** "이런 경험을 통해 AI 도구의 코드베이스 전체 탐색 활용에 관심을 갖게 되었다" 식으로 연결.

## 5. 리팩토링 언급 여부 판단

"안 한 이유를 설명하는 것" 자체가 **"해야 하는데 안 했다"는 인상**을 줄 수 있음.

**판단 순서:**
1. 리팩토링이 실제로 필요한 구조인가? → 선형적이고 읽기 쉬운 순수 계산 함수라면 필요 없을 수 있음
2. 필요 없다면 → 언급 자체를 빼기
3. 필요하지만 안 했다면 → 주된 이유(안정적으로 동작, 일정 대비 우선순위)를 먼저 쓰고, 부가적 이유는 보강 역할로

**"사용처가 한정되어 있어서"만** 쓰면 "그러면 오히려 합치기 쉬웠던 거 아닌가?"라는 반문이 가능하므로 주의.

## 6. 부정확한 표현 걸러내기

| 틀린 표현 | 맞는 표현 | 이유 |
|-----------|-----------|------|
| 구조화 | 리팩토링 | 구조가 없는 걸 새로 잡는 것 vs 이미 동작하는 걸 정리하는 것 |
| 사용처가 한정 → 리팩토링 안 함 | 안정적 동작 + 일정 대비 우선순위 낮음 | 한정되어 있으면 오히려 합치기 쉬운 조건 |

## 7. 재발(2차 수정) 이야기를 넣을지 판단

넣으면 좋은 경우: "전체 점검을 하지 않아서 놓친" 패턴 → 교훈이 명확
넣으면 나쁜 경우: 실은 본인이 다른 작업에서 같은 실수를 직접 반복한 것 → "같은 실수를 반복한 사람"이 됨

**이 케이스**: 2차 수정의 vatRate 버그는 1차 수정 이후 다른 커밋(`85ac0207`)에서 본인이 직접 만든 것. "놓쳤다"가 아니라 "반복했다"이므로 굳이 언급하면 약점이 됨. 1차 수정만으로 스토리가 충분하면 2차는 빼기.

## 8. 프로젝트 소재 선정 기준

- 사용자가 실제로 많이 쓰는 기능인지 (그룹화/프리셋은 사용률 낮아서 제외)
- About과의 일관성이 있는지 (UX 고민, AI 활용)
- "왜→판단→결과" 흐름을 솔직하게 쓸 수 있는지
- 효과가 제한적이었던 것도 있는 그대로 쓸 수 있는지
