## 답변 원칙

- **기능을 만들었다**보다 **왜 이 작업이 필요했고, 이후 무엇이 쉬워졌는지**를 먼저 말한다.
- 과장하지 않고, **내가 직접 판단하고 바꾼 부분**과 **팀/대표 제안으로 시작된 부분**을 구분해서 말한다.
- 숫자가 불확실하면 억지로 수치화하지 말고, **변경 비용 감소 / 검증 속도 향상 / 반복 작업 감소**처럼 정성적으로 답한다.
- 답변 순서는 가능하면 `배경 → 내가 한 일 → 좋아진 점 → 한계/배운 점` 순서로 간다.
- 아래 근거는 현재 로컬 코드베이스와 git history 기준이다. 프롬프트 히스토리까지는 가지 않아도 설명 가능한 근거가 충분했다.

## 1. Feature Gating / 요금제 구조 개편

### 왜 했어요?

페이지마다 요금제 조건과 온보딩 분기가 흩어져 있으면 신규 요금제나 정책이 들어올 때 여러 화면을 같이 고쳐야 했습니다. 실제로 결제/요금제 개편 시기에 레거시와 신규 요금제가 섞이면서, 기존 방식으로는 조건 누락이나 호환성 문제가 계속 생길 가능성이 컸습니다. 그래서 요금제 접근 제어를 페이지 로직이 아니라 공통 규칙으로 끌어올리는 게 먼저라고 판단했습니다.

### 뭐가 좋아졌어요?

- 기능 접근 정책을 한 곳에서 관리할 수 있게 됐습니다.
- 페이지마다 인라인 조건을 반복하지 않아도 돼서 변경 비용이 줄었습니다.
- 레거시 / 신규 요금제를 같은 매트릭스 안에서 다룰 수 있어 호환성 대응이 쉬워졌습니다.
- 테스트도 `evaluateFeatureAccess` 같은 순수 함수 단위로 검증할 수 있게 됐습니다.

### 면접에서 짧게 답하면

> 요금제 조건이 여러 페이지에 흩어져 있으면 신규 플랜이 들어올 때마다 화면별로 인라인 분기를 다 수정해야 해서, 결국 변경 비용이 커집니다. 그래서 FeatureMatrix, evaluateFeatureAccess, PlanGuardRoute 같은 공통 구조로 접근 제어를 올려서 정책은 한 곳에서 관리하고 화면은 그 결과만 쓰게 바꿨습니다. 이후에는 요금제 변경 시 페이지를 다 뒤지기보다 매트릭스와 가드 로직 중심으로 수정할 수 있게 된 게 가장 컸습니다.

### 더 물어보면 이렇게

- 이 작업은 **새 기능 추가**보다 **구조를 먼저 정리한 리팩터링**에 가까웠습니다.
- 핵심은 “기능별 required plan”을 선언적으로 두고, 페이지 컴포넌트는 그 규칙을 소비만 하게 만든 것입니다.
- 완전히 행동 변경이 목적은 아니었고, **기존 정책을 유지하면서 구조를 바꾼 작업**이라고 말하는 게 맞습니다.

### 솔직하게 말할 포인트

- 사용자 입장에서는 큰 화면 변화가 없을 수 있습니다.
- 대신 팀 입장에서는 정책 변경이나 신규 요금제 대응이 훨씬 쉬워졌다는 게 본질입니다.

## 2. Tailwind / 디자인 토큰 / 공통 UI 정리

### 왜 했어요?

스타일 토큰과 공통 UI가 정리되지 않은 상태에서는 화면을 만들 때마다 비슷한 스타일 판단을 반복하게 됩니다. 특히 MUI/Emotion과 sx 패턴이 넓게 퍼진 상태에서는 페이지마다 스타일 책임이 흩어지고, AI로 작업할 때도 어떤 패턴을 따라야 하는지 컨텍스트가 흐려졌습니다. 그래서 신규 코드와 정리하는 영역부터 Tailwind, 토큰, 공통 primitives 쪽으로 기준을 세우는 게 필요했습니다.

### 뭐가 좋아졌어요?

- 페이지마다 비슷한 스타일 결정을 반복하는 비용이 줄었습니다.
- `PageWrapper`, `Container`, `Loading`, `NoData` 같은 반복 UI를 재사용할 수 있게 됐습니다.
- 스타일 기준이 통일되면서 Figma 반영이나 UI 수정 속도가 더 안정적으로 나왔습니다.
- AI가 작업할 때도 참고해야 할 스타일 패턴이 더 명확해졌습니다.

### 면접에서 짧게 답하면

> 처음에는 화면을 빨리 만드는 데 집중했는데, 나중에는 스타일 기준이 흩어져 있는 게 더 큰 병목이라는 걸 느꼈습니다. 그래서 Tailwind와 디자인 토큰을 기준으로 정리하고, 반복되는 PageWrapper, Loading, NoData 같은 UI를 공통 컴포넌트로 묶었습니다. 결과적으로 화면별 중복 구현이 줄었고, 이후 스타일 수정이나 신규 페이지 작업이 더 예측 가능해졌습니다.

### 더 물어보면 이렇게

- 이 작업은 “전면 교체”보다는 **점진적 전환**에 가깝습니다.
- MUI/Emotion을 한 번에 다 걷어내는 것보다, **신규 코드와 정리 구간부터 기준을 바꾸는 방식**을 택했습니다.
- 핵심은 예쁘게 바꾸는 게 아니라, **반복 비용과 컨텍스트 비용을 줄이는 것**이었습니다.

### 솔직하게 말할 포인트

- MUI를 완전히 제거한 건 아닙니다.
- 하지만 Tailwind-first / token-first 기준을 세우고, 이후 작업 방향을 바꿨다는 데 의미가 있습니다.

## 3. mock data center / 분기 상태 재현 / AdsDiagnosis 구조 분리

### 왜 했어요?

결제나 광고 진단처럼 상태 분기가 많은 기능은 실제 데이터를 기다리거나 특정 조건을 다시 만들지 않으면 검증이 어렵습니다. 이 상태에서는 퍼블리싱, QA, 회귀 확인이 느려지고, AI로 작업할 때도 어떤 상태를 기준으로 고쳐야 하는지 매번 설명해야 했습니다. 그래서 분기 상태를 로컬에서 빨리 재현할 수 있는 환경과, AdsDiagnosis 자체의 구조 분리가 필요했습니다.

### 뭐가 좋아졌어요?

- 특정 상태를 다시 만들기 위해 운영 데이터나 백엔드 응답을 계속 기다리지 않아도 됐습니다.
- QA/회귀 테스트에서 “이 케이스를 다시 보는 비용”이 줄었습니다.
- AdsDiagnosis가 API / component / hook / modal로 분리되면서 기능 추가 범위를 파악하기 쉬워졌습니다.
- AI 채팅이나 코칭노트처럼 새 흐름이 붙을 때도 기존 구조 안에서 확장하기 수월해졌습니다.

### 면접에서 짧게 답하면

> 광고 진단이나 결제는 상태 분기가 많아서 한 번 기능을 붙인 뒤에 검증 비용이 더 크게 느껴졌습니다. 그래서 mock data center와 분기 재현 도구를 정리해서 특정 케이스를 로컬에서 바로 재현할 수 있게 했고, AdsDiagnosis도 API / hook / modal 단위로 나눠서 이후 확장 범위를 명확하게 만들었습니다. 결과적으로 기능 추가보다 검증과 회귀 확인이 쉬워진 게 가장 컸습니다.

### 솔직하게 말할 포인트

- 모든 상태를 완벽하게 자동화한 건 아닙니다.
- 하지만 자주 확인해야 하는 조합을 빠르게 재현할 수 있게 만든 것만으로도 개발 속도 차이가 컸습니다.

## 4. Claude page scaffold command

### 왜 했어요?

새 페이지를 만들 때마다 폴더 생성, 기본 파일 작성, 라우터 등록, 메뉴 등록 같은 기초 작업이 반복됐습니다. 이건 직접 해도 어렵지는 않지만 매번 챙겨야 하는 항목이 많아서, AI가 놓치거나 사람이 빠뜨리기 쉬운 영역이었습니다. 그래서 코드 구조 전체를 AI-friendly하게 다시 설계하기보다, 생성 규칙 자체를 명령으로 고정하는 게 더 확실한 해결책이라고 봤습니다.

### 뭐가 좋아졌어요?

- 페이지 시작 비용이 줄었습니다.
- 라우터/메뉴 등록 누락 같은 반복 실수가 줄었습니다.
- 프로젝트 컨벤션에 맞는 기본 구조를 일관되게 생성할 수 있게 됐습니다.
- 비슷한 유형의 페이지를 더 빠르게 열 수 있었습니다.

### 면접에서 짧게 답하면

> 새 페이지를 만들 때 어려운 건 구현보다 기초 세팅이었습니다. 폴더 구조, 라우터, 메뉴, 기본 컴포넌트 파일을 매번 맞추는 게 번거롭고 누락도 생길 수 있었어요. 그래서 코드베이스 전체를 다시 뜯기보다, 우리 컨벤션에 맞는 페이지를 명령으로 생성하는 scaffold를 만들었고, 이후에는 시작 작업을 구조적으로 줄일 수 있었습니다.

### 더 물어보면 이렇게

- “왜 AI가 기존 코드만 보고 하게 두지 않았냐?”라는 질문에는, 그 방향은 **코드베이스 자체를 더 큰 비용으로 정비해야 했고**, 당시엔 scaffold가 더 빠르고 확실한 선택이었다고 답하면 됩니다.
- 핵심은 AI를 똑똑하게 만드는 게 아니라, **AI가 실수하지 않는 구조를 만드는 것**이었습니다.

### 솔직하게 말할 포인트

- scaffold가 모든 페이지를 완성해주는 건 아닙니다.
- 대신 “반복되는 초기 작업”을 줄여주고, 시작 지점을 컨벤션에 맞게 맞춰주는 역할이었습니다.

## 5. GTM / 트래킹 구조 분리

### 왜 했어요?

기존에는 Mixpanel, Hotjar, 해싱 함수, GTM 코드가 `trackingHelper.ts` 안에 같이 들어 있었습니다. 당장 못 고칠 정도는 아니었지만, 새 이벤트가 늘어날수록 성격이 다른 로직이 한 파일에 계속 누적되는 구조였고, 특히 AI로 작업할 때는 어떤 패턴을 따라야 하는지가 흐려졌습니다. 그래서 GTM 쪽은 `shared/tracking` 구조로 분리해 관심사를 명확히 나누는 게 낫다고 판단했습니다.

### 뭐가 좋아졌어요?

- 새 이벤트 추가 패턴이 단순해졌습니다.
- 이벤트 상수 / 타입 / 훅 구조가 생겨서 실수 지점이 줄었습니다.
- 광고 분석이나 광고 진단처럼 새 도메인의 트래킹을 붙일 때 trackingHelper를 계속 건드리지 않아도 됐습니다.
- AI가 작업할 때도 한 파일 안에서 여러 관심사를 구분할 필요가 줄었습니다.

### 면접에서 짧게 답하면

> trackingHelper에 여러 분석 도구와 GTM 로직이 같이 들어 있으면 사람은 어떻게든 수정할 수 있어도, AI는 어떤 패턴을 따라야 하는지 헷갈리기 쉬웠습니다. 그래서 GTM 쪽을 shared/tracking으로 분리하고, 상수/타입/훅 구조로 정리해서 새 이벤트를 추가할 때 같은 패턴 안에서 작업하게 만들었습니다. 이후에는 광고 분석 쪽 GA4 이벤트를 붙일 때도 구조 안에서 일관되게 추가할 수 있었습니다.

### 솔직하게 말할 포인트

- 인간 입장에서는 파일 하나에 모여 있는 게 더 빠를 수도 있습니다.
- 이 작업은 “코드를 예쁘게 정리했다”보다 **AI와 반복 작업 기준에서는 이 구조가 더 낫다**는 판단에 가까웠습니다.

## 6. AdsDiagnosis / AI 채팅 / 코칭노트

### 왜 했어요?

광고 영역은 단순히 지표를 보여주는 데서 끝나면 사용자가 다음 행동으로 이어가기 어려웠습니다. 그래서 분석 → 캠페인 → 진단으로 흐름을 넓히고, 진단 결과를 보고 바로 액션으로 이어질 수 있게 AI 채팅과 코칭노트까지 붙이는 방향으로 확장했습니다.

### 뭐가 좋아졌어요?

- 단순 대시보드가 아니라 “문제 확인 → 해석 → 다음 액션” 흐름이 생겼습니다.
- 광고 성과를 숫자로만 보는 것이 아니라, 사용자 행동으로 이어질 수 있는 구조가 됐습니다.
- 이후 코칭노트 적용, AI 채팅 개선, 온보딩 보강 같은 확장도 같은 축 위에서 이어갈 수 있게 됐습니다.

### 면접에서 짧게 답하면

> 광고 기능을 단순히 캠페인 성과를 보여주는 대시보드로 두기보다, 문제를 해석하고 다음 액션까지 이어지게 하고 싶었습니다. 그래서 분석, 캠페인, 진단 흐름을 만들고, 진단 결과에서 바로 AI 채팅과 코칭노트로 이어지는 경험까지 붙였습니다. 사용자가 ‘광고가 안 좋다’에서 멈추지 않고 다음 행동으로 연결될 수 있게 한 게 핵심이었습니다.

## 7. 결제 / 추천인 / 쿠폰 / 혜택관리

### 왜 했어요?

결제는 단순 결제 성공/실패 화면만 있는 영역이 아니라, 실제로는 요금제 변경, 구독 유지/해지, 추천인, 쿠폰, 혜택관리 같은 후속 경험이 붙어야 사용자 흐름이 완성됩니다. 그래서 결제 이후의 사용자 동선까지 확장하는 방향으로 작업했습니다.

### 뭐가 좋아졌어요?

### 면접에서 짧게 답하면

## 8. 현금 흐름 캘린더

## 10. vendorId / DTO / SubAccountRequest / originVendorId

### 왜 했어요?

프론트에서 기능을 붙이려면 결국 필요한 데이터가 API 응답에 있어야 하고, 벤더 연결/광고 계정 연동처럼 도메인 규칙이 많은 영역은 프론트에서 우회 처리만으로는 한계가 있습니다. 그래서 필요한 경우 DTO, entity, 응답 필드, 요청 구조까지 같이 맞추는 쪽이 더 빠르고 정확하다고 봤습니다.

### 뭐가 좋아졌어요?

- 프론트에서 우회 로직을 덜 짜도 됐습니다.
- 같은 데이터를 프론트/백엔드가 다른 의미로 해석하는 문제가 줄었습니다.
- 광고 계정 연동, 벤더 매칭, 상태 조회 같은 기능의 정확도를 높일 수 있었습니다.

### 면접에서 짧게 답하면

> 프론트에서 필요한 데이터가 응답에 없으면 결국 화면 쪽 우회 로직만 늘어납니다. 그래서 광고 진단/연동 쪽에서는 vendorId 같은 응답 필드를 직접 추가하고 DTO와 도메인 모델도 같이 맞췄습니다. SubAccountRequest나 originVendorId 같은 데이터도 프론트에서 필요한 흐름 기준으로 함께 정리해서, 기능을 억지로 붙이기보다 구조 자체를 맞추는 쪽으로 일했습니다.

### 더 물어보면 이렇게

- `originVendorId`는 같은 업체의 WING / GROWTH / AD 계정을 묶어 해석하기 위한 기준점으로 설명하면 됩니다.
- `SubAccountRequest`는 단순 CRUD라기보다 **광고 계정 연동 흐름을 제품 안에서 다루기 위해 필요한 구조**였다고 말하는 게 좋습니다.

### 솔직하게 말할 포인트

- 백엔드 전체 설계를 내가 주도했다고 말하면 과장입니다.
- 대신 **프론트에서 필요한 흐름을 맞추기 위해 엔티티/DTO/API 응답까지 직접 건드린 경험**이라고 말하는 게 맞습니다.

## 11. DataGrid 전체화면 / 컬럼 접기

## 13. 비즈니스 마스터 (가설 검증용 분석 리포트)

### 이 작업의 성격

이건 운영에 정식으로 크게 딜리버리된 핵심 기능이라기보다, 대표님과 함께 진행한 **가설 검증용 분석 리포트 작업**에 가깝습니다. 대표님이 엑셀 기반으로 보고 있던 대시보드성 자료를 화면으로 옮겨보면서, 실제로 서비스 안에서 이런 분석 리포트가 의미가 있는지 빠르게 검증해보는 성격이었습니다.

### 왜 했어요?

핵심은 “새로운 정식 제품 기능을 내가 기획했다”가 아니라, 대표님이 엑셀로 보고 있던 분석 가설을 실제 UI로 빠르게 옮겨서 검증하는 것이었습니다. 매출/광고/비용/상품 4개 관점의 리포트를 화면으로 붙여보면, 사용 흐름이나 데이터 표현 방식에서 무엇이 필요한지 더 빨리 볼 수 있다고 판단했습니다.

### 내가 한 역할은 어디까지였어요?

- 대표님이 엑셀 형태의 대시보드 자료와 가설을 주셨고
- 나는 그걸 **UI로 빠르게 구현**해서 실제 화면에서 확인할 수 있게 만들었습니다.
- 이 과정에서 백엔드는 기존 REST 흐름을 그대로 재사용하기보다, **내가 직접 GraphQL을 붙여서** 필요한 데이터를 가져오는 방식을 택했습니다.
- 즉 제품 방향이나 비즈니스 가설 자체를 내가 만든 건 아니고, **그 가설을 실험 가능한 UI/데이터 흐름으로 구현한 역할**이라고 설명하는 게 맞습니다.

### 뭐가 의미 있었어요?

- 며칠 안에 기초 형태를 빠르게 붙여볼 수 있었습니다.
- 엑셀에서 보던 가설을 실제 서비스 화면으로 옮기면서, 어떤 리포트가 유효한지 더 빠르게 검토할 수 있었습니다.
- 기술적으로는 Apollo / GraphQL을 직접 붙이면서 별도 분석 리포트용 데이터 흐름을 만든 경험이 있었습니다.
- 다만 운영에 강하게 노출된 최종 기능이라기보다, **가설 검증용 실험 페이지**라는 점은 분명히 선을 긋고 말하는 게 맞습니다.

### 면접에서 짧게 답하면

> 비즈니스 마스터는 대표님이 엑셀로 보시던 분석 가설을 실제 화면으로 빠르게 옮겨보는 실험 성격의 작업이었습니다. 매출, 광고, 비용, 상품 4개 관점의 리포트를 UI로 구현했고, 데이터도 기존 흐름을 그대로 쓰기보다 제가 Apollo/GraphQL을 직접 붙여서 가져왔습니다. 다만 이걸 제가 제품적으로 기획했다고 말하는 건 맞지 않고, 가설을 검증할 수 있는 UI와 데이터 흐름을 빠르게 구현한 역할이라고 설명하는 게 정확합니다.

### 솔직하게 말할 포인트

- “서비스에 크게 딜리버리된 메인 기능”처럼 말하면 과장입니다.
- “대표와 함께 가설 검증을 진행한 실험성 작업”이라고 말하는 게 맞습니다.
- 의미는 사업 임팩트보다, **빠르게 가설을 화면으로 옮기고 데이터를 붙여본 실행 경험**에 있습니다.

## 0. 가설검증 속도 10배 — 어떻게 설명할 것인가

### 물어보면

> 이력서 제목에 “가설검증 속도를 최대 10배 끌어올렸다”고 적혀 있는데, 근거가 뭐예요?

### 짧게 답하면

> 입사 초기에 신규 페이지 하나를 만드는 데 약 10일 걸렸습니다. 기능을 만들면서 페이지 템플릿, 공통 컴포넌트, 요금제 접근 규칙 구조, 스타일 토큰, 페이지 스캐폴딩 도구를 함께 정리해왔고, scaffold와 공통 UI가 완비된 이후에는 같은 성격의 페이지를 하루 안에 붙여볼 수 있게 됐습니다. 10일에서 1일이니까 최대 10배입니다.

### 더 물어보면 이렇게

- “최대”라고 쓴 이유는, 모든 페이지가 하루 만에 끝나는 건 아니기 때문입니다. AdsDiagnosis는 약 2일, ProductsProfit은 약 4일이었고, scaffold 이후 가장 빠른 케이스 기준이 하루입니다.
- 기획 논의나 디자인 대기 시간은 포함하지 않았고, 순수 FE 구현 기준입니다.
- 경력기술서에 시점별 속도 테이블이 있어서 근거를 보여드릴 수 있습니다.

### 솔직하게 말할 포인트

- 10배는 **가장 빠른 케이스 기준**이고, 평균적으로는 3~5배 수준입니다.
- 손이 빨라졌다기보다, **시작 비용을 구조적으로 줄인 결과**라고 설명하는 게 정확합니다.
