## 2. 스토리 다듬기 과정에서 나온 피드백과 교훈

### 피드백 1: "왜 했는지가 중요하다"

처음에는 STAR 포맷으로 아키텍처 설명 위주로 작성했는데, 실제로 중요한 건:
- 이 작업을 시간 써서 할 만큼 가성비가 있었는지
- 작업 전후로 실제로 어떤 변화가 있었는지

→ Git 히스토리로 구체적 수치를 뽑아서 근거를 만듦

### 피드백 2: "버그 강조는 빼라"

버그 4건 → 0건을 강조했었는데, 버그는 단순 구현 실수였지 구조 문제는 아니었음.
- ❌ "버그가 4건 터졌다" → 구조 때문이 아닌 단순 실수를 과장하는 꼴
- ✅ "디버깅이 힘들고 수정·추가가 번거로웠다" → 이것도 과장이라는 피드백

### 피드백 3: "과장하지 마라"

"영향 범위를 파악하기 힘들었다", "추적하기 어려웠다" 같은 표현이 과장이라는 지적.
실제로는:
- 추가: 맨 아래에 새 함수 붙이면 됨
- 수정: 특정 함수 찾아서 고치면 됨
- 다만 다양한 도메인 이벤트가 계속 쌓이는 게 부담스러웠던 것

→ "당장 못 하는 건 아니지만 방향이 맞나 의문" 정도의 톤이 솔직함

### 피드백 4: "내 말을 그대로 붙이지 말고 네가 생각해서 정리해라"

사용자의 구어체 설명을 거의 그대로 옮겨적으면 안 됨. 핵심을 이해하고 재구성해야 함.

### 피드백 5: "인간보다 AI한테 좋은 구조"

실제 코드를 확인한 결과:
- `useAdAnalysisTracking.ts`: 30개 함수가 전부 동일한 형태
- `trackingHelper.ts`: Mixpanel SDK 초기화, Hotjar 설정, crypto 해싱, GTM 코드가 혼재

**솔직한 판단**: 인간 입장에서는 파일 하나에 모여있는 게 스크롤만 하면 되니까 오히려 편할 수 있음. 분리하면 constants, types, 훅 파일 3개를 왔다갔다 해야 해서 번거로운 면도 있음. 하지만 AI 입장에서는 관심사가 분리된 구조가 판단할 게 없어져서 정확도가 올라감.

→ "코드를 깔끔하게 정리했다"가 아니라 "AI가 정확하게 작업할 수 있는 환경을 만들었다"가 포인트

### 피드백 6: "흐름이 중요하다"

결과(GA4 이벤트 추가 시 체감)가 중간에 나오면 안 되고, 아래 순서로 흘러야 함:
1. **상황** — trackingHelper가 만능 파일이 되어가고 있었음
2. **판단** — 이대로 가면 안 되겠다, 특히 AI로 작업할 때 문제가 될 거다
3. **전략** — 새 구조 설계 + 점진적 이관
4. **결과** — 실제로 GA4 이벤트 대량 추가할 때 체감

## 3. 최종 스토리

GTM 트래킹을 도입할 때 기존에 Mixpanel, Hotjar, GA 등 분석 도구를 모아둔 trackingHelper.ts에 코드를 추가했음. 이미 600줄이 넘는 파일이었는데, 여기에 회원가입·결제·쇼핑몰 연동 같은 GTM 이벤트까지 들어가면서 성격이 다른 도메인 로직들이 하나의 파일 안에 계속 누적되는 구조가 됨. 당장 작업이 안 되는 건 아니었지만, 광고 분석 쪽 트래킹이 본격적으로 붙을 예정이었고 그때마다 이 파일이 끝없이 커지는 방향이 맞는 건지 의문이 들었음.

특히 AI로 트래킹 코드를 작업할 때가 걸렸음. trackingHelper에는 Mixpanel 초기화, Hotjar 설정, 해싱 함수 등이 GTM 코드와 같은 파일에 섞여있어서, AI한테 컨텍스트로 넘기면 어떤 패턴을 따라야 하는지 혼동이 생겼음. 사실 인간 입장에서는 파일 하나에 다 모여있는 게 스크롤만 하면 되니까 오히려 편할 수도 있는데, AI한테는 관심사가 섞여있는 파일이 오히려 방해가 됨. GA4 이벤트만 다루는 구조로 분리하면 AI가 판단할 게 없어지니까 정확도가 올라갈 거라고 생각했음.

그래서 GTM 도입 초기에 shared/tracking/ 이라는 구조를 따로 설계했음. GTM 통신은 하나의 클라이언트로 격리하고, 도메인별로 커스텀 훅을 분리해서 이벤트 파라미터에도 타입을 걸어두는 방식이었음. 기존 코드를 한꺼번에 옮기는 대신, 새로 추가하는 트래킹만 이 구조로 작성하고 기존 코드는 나중에 여유가 될 때 이관하는 전략을 택함.

실제로 효과를 체감한 건 광고 분석 GA4 이벤트 15개를 추가할 때였음. 분리된 훅 파일에는 동일한 형태의 함수만 있으니까 AI가 그 패턴대로 하나 더 만들면 끝이었음. 이후 광고 진단 기능을 집중 개발하면서도 trackingHelper를 건드릴 일이 없었고, 별도의 리팩토링 일정 없이 기능 개발 흐름 안에서 자연스럽게 이관을 마무리했음. 다만 Mixpanel/Hotjar 같은 레거시 도구는 아직 trackingHelper에 남아있는 상태임.

## 4. 이력서 스토리 작성 시 얻은 교훈

- **과장은 역효과**: 실제로 안 힘들었던 걸 힘들었다고 쓰면 면접에서 파고들었을 때 무너짐
- **솔직한 트레이드오프가 신뢰를 줌**: "인간한테는 오히려 불편할 수 있다"를 인정하면서 "AI한테는 이게 맞다"고 말하는 게 더 설득력 있음
- **흐름이 중요**: 상황 → 판단 → 전략 → 결과 순서로 읽히게 해야 함
- **톤 통일**: 구어체로 가되, 내용은 논리적으로 구성
