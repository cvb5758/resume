# GTM 트래킹 아키텍처 분리

## 상황 (Situation)

이커머스 SaaS 플랫폼의 사용자 행동 분석을 위해 GTM(Google Tag Manager) 기반 이벤트 트래킹을 도입해야 했다. 기존에는 Mixpanel, Hotjar, Google Analytics, Facebook Pixel 등 여러 분석 도구가 `trackingHelper.ts` 한 파일(600줄+)에 모두 들어 있었고, 여기에 GTM 코드까지 추가해야 하는 상황이었다.

## 문제 인식 (Task)

GTM을 처음 구현할 때 기존 패턴대로 trackingHelper에 넣었지만, 이틀간 작업하면서 문제가 명확해졌다.

- 회원가입, 결제, 쇼핑몰 연동 등 **도메인이 다른 트래킹 로직이 한 파일에 계속 쌓이고 있었다**
- GTM 도입 직후 트래킹 관련 **버그 수정이 4건 연속 발생**했다 (파라미터 누락, 포맷 오류, 이전 이벤트 데이터 오염 등)
- 향후 광고 분석 등 새로운 도메인의 트래킹이 대량 추가될 예정이었다
- 파라미터 타입이 느슨해서 **런타임에서야 잘못된 데이터가 발견**되고 있었다

단순히 "코드가 길어서" 분리한 게 아니라, **트래킹 버그가 반복 발생하는 구조적 원인이 이 파일에 있다**고 판단했다.

## 행동 (Action)

### 점진적 마이그레이션 전략

한 번에 전체를 리팩토링하는 대신, GTM 도입 2일 차에 `shared/tracking/` 구조를 설계하고 **새 기능은 처음부터 새 구조로, 기존 코드는 나중에 이관**하는 전략을 택했다.

**1단계 — 구조 설계 및 새 기능 적용 (2025.01)**

```
shared/tracking/
├── api/gtm-client.ts        # GTM dataLayer 푸시 (단일 부수효과 채널)
├── constants/constants.ts   # 이벤트 카테고리/이름 상수
├── model/types.ts           # 이벤트별 파라미터 타입 정의
├── hooks/                   # 도메인별 커스텀 훅
└── index.ts
```

- GTM 데이터 레이어 통신을 `gtm-client.ts` 하나로 격리
- 각 도메인 이벤트를 **커스텀 훅**으로 캡슐화 → 컴포넌트는 훅이 제공하는 함수만 호출
- 이벤트 파라미터에 **TypeScript 타입을 정의**해서 잘못된 데이터를 컴파일 타임에 잡도록 설계
- 계산기 앱 트래킹을 시작으로, 새로 추가되는 트래킹은 모두 이 구조로 작성

**2단계 — 기존 코드 이관 (2026.02)**

광고 분석 GA4 이벤트 15개가 추가되고, 광고 진단(AdsDiagnosis) 트래킹까지 확장된 시점에서 trackingHelper에 남아있던 기존 GTM 코드를 일괄 이관했다.

- 회원가입, 결제, 쇼핑몰 추가 등 기존 GTM 함수를 도메인별 훅으로 변환
- trackingHelper에서 GTM 관련 코드 215줄 제거
- `adsDiagnosisAnalytics.ts`(162줄) 등 페이지에 산재된 트래킹 유틸도 통합
- trackingHelper는 Mixpanel/Hotjar 등 레거시 도구 전용으로 역할 축소

## 결과 (Result)

### 트래킹 버그 제로

- 분리 전: GTM 도입 직후 **트래킹 관련 버그 수정 4건** (파라미터 오류, 데이터 오염, Hotjar 에러 등)
- 분리 후: 트래킹 관련 **버그 수정 0건**

TypeScript 타입 정의가 파라미터 실수를 컴파일 타임에 잡아주면서, 런타임 트래킹 오류가 사라졌다.

### 기능 개발 속도 향상

분리 이후 광고 진단(AdsDiagnosis) 기능을 집중적으로 개발하면서 **19건의 커밋**이 이어졌는데, 이 과정에서 trackingHelper를 **한 번도 수정하지 않았다.** 도메인별 훅 구조 덕분에 각 기능 개발자가 자신의 영역만 수정하면 되는 구조가 되었다.

### 최소 비용의 점진적 전환

- 최종 이관 커밋은 **+816 / -454줄** 규모로, 반나절 작업량
- 기존 서비스 중단 없이 1년에 걸쳐 점진적으로 완료
- 새 기능 개발과 리팩토링을 **동시에** 진행할 수 있는 전략이었기 때문에 별도의 리팩토링 스프린트가 필요하지 않았다

### 확장 가능한 패턴 확립

최종적으로 60개 이상의 이벤트 상수, 50개 이상의 파라미터 타입, 25개의 도메인별 훅이 `shared/tracking/`에 체계적으로 정리되었고, 새 트래킹 이벤트를 추가할 때 **상수 정의 → 타입 정의 → 훅 작성**이라는 일관된 패턴이 만들어졌다.
