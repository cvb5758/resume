# 이력서 소재 정리

## 프로젝트 개요

**장사왕 (이커머스 관리 SaaS) — 프론트엔드 개발 (1년 10개월, 1인 담당)**

1인 프론트엔드로 매출/정산 분석, 매입원가 관리, 상품별 순이익, 쿠팡 광고 분석 등 주요 기능 개발과 성능 최적화, DataGrid 고도화, 로딩 UX 개선을 담당.

---

## 소재 1: DataGrid 전체화면 기능

### 왜 했는지

주문 상세 분석 페이지는 컬럼이 20~30개에 달하는데, 사이드바 + 검색영역을 제외하면 실제 그리드 영역이 좁아서 가로 스크롤이 과도했음. "데이터를 넓게 보고 싶다"는 사용자 요구가 있었음.

### 리소스 대비 효과 판단

전체화면은 기존 그리드 위에 래핑만 추가하면 되니 구현 비용이 낮고, 데이터 집중형 페이지(주문목록, 주문상세, 상품별 순이익)에 공통 적용 가능해서 투자 대비 활용도가 높다고 봄.

### 실제 한 일

- 1차: MUI Dialog(fullScreen) 안에 새 그리드를 렌더링하는 방식으로 프로토타입
- 2차: ToolBar에 전체화면 버튼 통합, 커스텀 SVG 아이콘, 3개 페이지 적용
- 3차: Dialog 방식의 문제(그리드 2개 존재 → 메모리, 상태 불일치, 깜빡임) 발견 → CSS `position: fixed` 방식으로 전환하여 **기존 그리드 그대로 확장**

### 결과 — 솔직하게

최종 방식(CSS fixed)이 코드량도 적고(-60~80줄) 상태 일관성도 확보됨. 처음부터 이 방식으로 갔으면 시행착오가 줄었을 텐데, Dialog 방식을 먼저 시도한 건 MUI 패턴에 익숙해서 관성적으로 선택한 측면이 있음. 결과적으로 올바른 방향을 찾았지만 과정이 효율적이진 않았음.

## 소재 2: 매입원가 관리 0원 입력 불가 버그 수정

### 이력서 문구

매입원가에 0을 입력하면 미입력(`-`)으로 표시된다는 버그 리포트를 받음. 무료 샘플 등 매입가가 실제로 0원인 상품을 관리할 수 없는 상태였음.

원인을 추적해보니 단일 지점 문제가 아니었음. 프로젝트 공통 유틸 함수가 숫자 0을 "빈 값"으로 판정하고 있었고, 이 함수를 그리드의 모든 숫자 포매터가 사용하고 있어서 0이 입력된 셀이 전부 `-`로 표시됨. 거기에 매입원가 계산 로직에서도 JavaScript의 falsy 특성(`!0 === true`, `0 || 기본값 === 기본값`)을 고려하지 않은 패턴이 입력값 변환, 부가세율 초기화, 합산 계산 등 여러 곳에 걸쳐 있었음.

모든 null 체크를 `=== null || === undefined`로 교체하고, 미입력 상태와 0 입력을 구분할 수 있도록 값 변환 로직을 정비함. 같은 falsy 패턴이 계산 로직 곳곳에도 있어서 순이익·마진율 계산에도 영향이 있을 수 있다고 판단해 함께 수정했지만, 실제로 계산 결과가 어디까지 왜곡되고 있었는지를 정밀하게 검증하지는 못했음.

계산 함수의 리팩토링은 검토했지만, 계산 로직이 안정적으로 동작하고 있고 사용처도 한정되어 있어서, 기능 개발 일정 대비 리팩토링의 우선순위가 낮다고 판단함.

### 기술 상세 (면접 대비)

#### 원인 1: isEmpty() 유틸 함수

```typescript
// src/utils/ObjectUtil.ts
export const isEmpty = (value: any) => {
  if (value == '' || value == null || value == undefined || value == 0 || ...) {
    return true;
  }
};
```

`value == 0` 조건이 명시적으로 들어가 있어서, 숫자 필드에서 0을 "빈 값"으로 판정.

#### 원인 2: NumberFormatter

```typescript
// Before
return isEmpty(params.value) ? '-' : Number(params.value).toLocaleString() + suffix;

// After
if (params.value === null || params.value === undefined || params.value === '') {
  return '-';
}
return Number(params.value).toLocaleString() + suffix;
```

#### 원인 3: calculateValue() 입력값 변환

```typescript
// Before — falsy 패턴 + indexOf 버그
const newData = excludedKeys.indexOf(key) > 0  // ← index 0번 누락
  ? selectedVal
  : selectedVal || selectedVal == 0 ? Number(selectedVal) : '';

// After
if (excludedKeys.indexOf(key) > -1) {
  newValue[key] = selectedVal;
} else if (selectedVal !== null && selectedVal !== undefined && selectedVal !== '') {
  newValue[key] = Number(selectedVal);
}
```

#### 원인 4: 매입가 미입력 시 강제 0 세팅

```typescript
// Before — 0 입력해도 다시 0으로 세팅 (미입력과 구분 불가)
if (isEmpty(newValue.purchasePrice)) {
  newValue.purchasePrice = 0;
}
// After — 삭제
```

#### 근본 원인 요약

| 패턴 | 왜 문제인가 |
|------|-------------|
| `isEmpty(value)` 에서 `value == 0` | 0을 명시적으로 빈 값 취급 |
| `!value` | JS falsy: `!0 === true` |
| `value \|\| default` | JS falsy: `0 \|\| default === default` |
| `indexOf(key) > 0` | 배열 첫 번째 요소(index 0) 누락 |

모두 JavaScript의 falsy 동작을 의식하지 못하고 "값이 없으면" 의미로 사용한 패턴. 숫자 0이 유효한 비즈니스 값인 도메인에서는 반드시 `=== null || === undefined`로 명시적 비교 필요.

#### 리팩토링하지 않은 이유

calc.ts에 3개 계산 함수(calculateValue, calcForMargin, calcForGlobalMargin)가 있지만:
- 제어 흐름이 단순 — 위에서 아래로 순서대로 계산하는 선형 구조
- 각 함수가 독립적인 순수 계산 — 입력 받고, 계산하고, 결과 반환
- "비슷하지만 다른" 계산기 — 공통 부분을 억지로 추출하면 분기 조건이 늘어나서 오히려 가독성 저하 가능

→ 리팩토링이 꼭 필요한 구조는 아님. 진짜 문제는 함수 구조가 아니라 falsy 패턴이었고, 그건 이미 수정함.

#### 백엔드 관련 (면접 질문 대비)

프론트의 0/null 버그와 동시기에 백엔드에서도 매입원가 관련 이슈 발생:
- purchasePrice upsert 오류 (기존 데이터 오류로 무조건 insert 임시 처리)
- purchaseYn(매입가 입력 여부) 판정 SQL 조건 버그
- 매입가 미입력 조회 시 vendorId 불일치, 판매가 없는 상품 오탐

프론트와 직접적 인과관계는 없지만, 매입원가 도메인 자체가 프론트/백 양쪽에서 null 처리가 복잡한 영역이었음.

## 소재 3: 매입원가 페이지 간 컨텍스트 연결

### 왜 했는지

매입원가 관리 페이지는 독립적으로만 접근 가능했음. 셀러가 실제로 매입가를 입력하고 싶은 순간은 "대시보드에서 마진이 이상하네?" "주문 상세에서 이 상품 원가가 빠져있네?" 같은 다른 페이지에서 문제를 발견했을 때. 매입원가 페이지로 직접 찾아가서 다시 상품을 검색해야 하는 동선이 비효율적이었음.

### 리소스 대비 효과 판단

이동 버튼 하나 + URL 파라미터 전달이라 개발 비용이 낮음. 하지만 5개 페이지에서 진입점을 만들어야 해서, 각 페이지의 데이터 구조(productId vs optionCode)에 따른 파라미터 변환 처리가 필요했음.

### 실제 한 일

- 5개 페이지(대시보드, 주문목록, 주문상세, 상품별 순이익, 광고캠페인)에 매입원가 이동 버튼 추가
- 쿠팡윙은 optionCode→productId 변환 후 전달, 그 외는 productId 직접 전달
- 이동 시 날짜, 쇼핑몰(vendorId), 캠페인(campaignId) 파라미터 함께 전달해서 자동 필터링
- 매입원가 페이지 진입 시 "미판매 포함 전체 상품" 기본 노출로 변경

### 결과 — 솔직하게

매입가 입력 경험이 "검색해서 찾아가기"에서 "발견 즉시 입력"으로 바뀜. 다만 마켓별 ID 체계가 달라서 변환 로직이 추가되었고, 처음에 업체코드(vendorId)가 파라미터에 빠져있어서 이동 후에도 쇼핑몰 선택이 안 되는 버그가 한 번 발생. 엣지 케이스를 초기에 전부 커버하지 못한 부분.

## 소재 4: 상품별 순이익 페이지 신규 개발

### 왜 했는지

기존에 셀러가 상품별 수익성을 확인하려면 주문 상세 분석에서 수백 건의 주문을 눈으로 훑거나, 엑셀로 뽑아서 피벗을 돌려야 했음. "이 상품이 진짜 돈 벌고 있는 건지"를 한눈에 보여주는 페이지가 없었음.

### 리소스 대비 효과 판단

백엔드에서 상품별 집계 API를 준비하고 있어서, 프론트는 Master-Detail 그리드 + 마진 상태 시각화에 집중하면 됨. ag-grid Enterprise의 Master-Detail 기능을 활용하면 상품→옵션 펼침을 직접 구현하지 않아도 되므로 공수 대비 효과가 괜찮다고 판단.

### 실제 한 일

- Master-Detail 그리드: 상품 행 클릭 → 옵션별 상세 펼침
- 마진 상태 시각화: 마진율에 따라 역마진/저마진/양호 태그 + 색상 표시
- 최소 ROAS 필드: 광고비 대비 최소 필요 매출 수치 제공
- WarningButton: 역마진 상품에 경고 아이콘으로 주의 유도
- 전체화면, 드래그 복사 붙여넣기, 헤더 Info 툴팁 등 DataGrid 공통 기능 적용

### 결과 — 솔직하게

셀러가 수익성을 상품 단위로 파악하는 화면이 생긴 것 자체가 의미 있었음. 다만 데이터 정확도가 매입가 입력 여부에 의존하는데, 매입가를 입력하지 않은 사용자에게는 순이익이 과대 계산됨. 이 부분을 UI에서 명확히 안내하지 못해서 "순이익이 높게 나오는데 실제로는 아닌" 상황이 발생할 수 있는 한계가 있음.

## 소재 5: 로딩 경험 체계 정비

### 왜 했는지

페이지 전환 시 화면 전체가 하얗게 비는 시간이 있었음. 특히 사이드바 메뉴 클릭 → 전체 화면이 GlobalLoading(스피너)으로 덮이면서 사용자가 "앱이 멈췄나?"라고 느끼는 상황이 반복됨. 로딩 방식이 페이지마다 제각각이라 일관성도 없었음.

### 리소스 대비 효과 판단

로딩 표시 자체는 소규모 UI 작업이지만, 모든 페이지 전환에 영향을 미치는 변경이라 체감 효과가 큼. 스켈레톤은 페이지별로 실제 레이아웃과 비슷하게 만들어야 해서 수작업이 필요했지만, 한번 만들어두면 유지보수 비용이 낮음.

### 실제 한 일

- NProgress 로딩 바 도입 → 페이지 전환 시 상단 얇은 프로그레스 바
- 사이드바 메뉴 클릭 시 전체화면 로딩(GlobalLoading) → 콘텐츠 영역만 로딩으로 변경
- 7개 주요 페이지에 스켈레톤 UI 구현
- loading.tsx 통합으로 로딩 패턴 일원화

### 결과 — 솔직하게

"화면이 비는" 느낌은 확실히 줄었음. 특히 사이드바가 유지된 채 콘텐츠만 전환되는 것이 SPA답게 느껴진다는 피드백이 있었음. 다만 스켈레톤이 실제 데이터 로딩 시간을 줄여주는 건 아니고 체감 속도만 개선하는 것이라, 근본적인 API 응답 속도 개선과 함께 가야 의미가 있음. 스켈레톤만으로는 느린 API를 숨기는 데 한계가 있다는 걸 느낌.

## 소재 6: 쿠팡 광고 분석 — 실험실에서 정식 메뉴까지

### 왜 했는지

장사왕은 주문/정산 분석이 핵심이었는데, 셀러들이 "광고비를 쓰고 있는데 실제로 수익이 나는 건지 모르겠다"는 피드백이 많았음. 특히 쿠팡 로켓그로스 셀러는 광고비가 매출에 직접 영향을 주는데, 이를 한 화면에서 볼 수 없었음.

### 리소스 대비 효과 판단

프론트 입장에서는 기존 DataGrid 인프라(전체화면, 정렬, 필터 등)를 재활용할 수 있어서 UI 개발 비용이 처음보다는 낮았음. 실험실(Lab)에서 먼저 오픈해서 검증 후 정식 메뉴로 이전하는 단계적 접근을 택함.

### 실제 한 일

- 광고 분석(AdsAnalysis): 캠페인별 광고비/매출/ROAS 대시보드, TrendChart 추이
- 캠페인 상세(AdsCampaign): 키워드별 매출, 상품별 광고 메트릭
- 광고 수익성 진단(AdsDiagnosis): 캠페인별 건강 상태(RED/YELLOW/GREEN) 진단
- 온보딩: 광고 계정 미연동 사용자를 위한 가이드 화면
- 실험실 → 정식 메뉴 이전: 오픈베타 제거, 메뉴 위치 변경, 모바일 메뉴 제외 처리

### 결과 — 솔직하게

광고 데이터를 기존 매출 데이터와 함께 볼 수 있게 된 것이 셀러에게 가치 있었음. 다만 쿠팡 광고만 지원하고 네이버/11번가 등은 미지원이라 일부 셀러에게는 "절반만 보이는" 상태. 또한 광고 진단(RED/YELLOW/GREEN)의 기준값이 데이터 기반이 아닌 도메인 전문가의 경험치로 설정되어 있어서, 정밀도에 대한 의문이 남아있음.

## 전체 작업 리스트 (포트폴리오용)

### 데이터를 더 빠르게 보여주기

- 앱 초기 로딩 시 tui-grid/ag-grid/toast-ui를 동기 로드하던 구조를 코드 스플리팅 + requestIdleCallback 비동기 초기화로 전환
- 페이지 전환 시 전체 화면이 비던 문제를 NProgress 로딩 바 + 콘텐츠 영역만 로딩 + 주요 7개 페이지 스켈레톤 UI로 개선
- Dashboard 워터폴 로딩 제거, Toast UI 에디터 지연 로딩, SDK 부트스트랩 비차단 처리

### 데이터를 더 편하게 다루게 하기

- 주문/상품 그리드에 전체화면 모드 도입 (Dialog 방식 → CSS fixed 방식으로 개선하여 그리드 이중 렌더링 제거)
- 그리드 컬럼 사이즈 자동 재계산(onGridSizeChanged)이 사이드바/전체화면 전환 시 불필요하게 동작하는 문제 제거
- 상품별 순이익 페이지 신규 개발 — Master-Detail 그리드로 상품→옵션 펼침, 마진 상태 시각화

### "발견 즉시 행동"할 수 있는 동선 설계

- 대시보드/주문목록/주문상세/상품순이익/광고캠페인 → 매입원가 관리로의 이동 버튼 + 컨텍스트(날짜, 쇼핑몰, 상품ID) 자동 전달
- 마켓별 ID 체계 차이(쿠팡 optionCode vs 네이버 productId) 변환 처리

### 데이터 정확성 확보

- 매입원가 관리에서 0 입력 시 미입력으로 표시되는 버그 수정 — 공통 유틸의 falsy 패턴이 숫자 0을 null과 동일하게 처리하는 근본 원인 파악 후 교체
- 매입원가 저장 시 그로스 입출고/배송비가 null로 전송되던 버그 수정

### 신규 기능 개발

- 쿠팡 광고 분석: 캠페인별 ROAS 대시보드, 키워드별 매출, 수익성 진단 — 실험실에서 검증 후 정식 메뉴로 단계적 이전
- 매입원가 로켓그로스 순이익 계산기: 카테고리 수수료, 입출고/배송 수수료, 반품률 반영
- 현금 흐름 캘린더: 정산 데이터 기반 월별 캘린더, 엔트리 CRUD, 반복 기능
