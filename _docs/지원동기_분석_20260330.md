## 1. 레브잇/올웨이즈 ★★★★★

### 지원 동기
> 이커머스 정산/순이익 분석 SaaS에서 1인 프론트엔드로 결제·상품·주문 등 제품 전 영역을 개발·운영하면서, 문제를 직접 발견하고 구조를 개선해왔습니다. AI를 활용하기 위해 코드 환경을 정돈하고, 스캐폴딩 도구를 Claude skills로 만들어 비개발자에게 인수인계하는 등 반복 작업을 자동화해왔습니다. 커머스 도메인과 AI 도구 활용 경험을 올웨이즈에서 이어갈 수 있다고 생각해 지원했습니다.

### 왜 이렇게 썼는가
- JD가 "커머스 도메인 경험", "AI 개발 툴 활용", "자동화로 팀 생산성 향상"을 우대사항으로 명시 → 세 가지 모두 이력서에 직접 근거가 있어서 구체적으로 풀어씀
- "Claude skills로 만들어 비개발자에게 인수인계" → JD에 Claude Code가 명시되어 있어서 도구명을 구체적으로 언급

### 핵심 매칭 근거
| JD 요구사항 | 이력서 근거 |
|------------|-----------|
| 커머스/플랫폼 도메인 경험 (우대) | 이커머스 정산/순이익 분석 SaaS |
| AI 개발 툴 활용 (우대) | About 전체 + 모든 프로젝트에 AI 활용 맥락 |
| Claude Code, Cursor 등 (기술 스택) | Claude skills로 스캐폴딩 구축 + 인수인계 |
| 자동화/최적화로 팀 생산성 (우대) | 스캐폴딩 도구, 데이터셋 구축 |
| 주도적 문제 발견 → 솔루션 제안 | 요금제 구조 개편, 초기 로딩 성능 발견 |
| 빠르게 개발하고 배포 | 1인 프론트엔드 전 영역 + "빠르게 만들어 시장 반응 확인" |
| React, TypeScript, Zustand, React Query | 동일 스택 |

### 면접에서 어필할 포인트
1. **커머스 도메인을 이미 알고 있다** — 결제·상품·주문·연동 등 이커머스 전 영역을 직접 만들어봤다
2. **AI 도구를 "잘 쓰기 위한 환경"을 만든 경험** — 단순 사용이 아니라, AI가 잘 작동하도록 코드 구조를 정돈한 관점
3. **자동화로 팀 생산성을 높인 구체적 사례** — 스캐폴딩 → Claude skills → 비개발자 인수인계
4. **문제를 직접 발견하고 해결한 패턴** — Hotjar에서 성능 문제 발견, 요금제 구조 문제를 스스로 판단

## 2. 인티그레이션/린다이어트 ★★★★☆

### 지원 동기
> 이커머스 SaaS에서 1인 프론트엔드로 결제·상품·주문 등 제품 전 영역을 개발·운영하면서, Hotjar 녹화에서 사용자 행동을 확인해 성능을 개선하는 등 문제를 직접 발견하고 해결해왔습니다. AI를 활용하기 위해 코드 환경을 정돈하고, 스캐폴딩 도구를 Claude skills로 만들어 비개발자에게 인수인계한 경험이 있습니다. 매주 새로운 기능이 출시되는 환경에서 빠르게 만들고 검증하는 방식이 그동안의 경험과 이어지는 부분이라 지원했습니다.

### 왜 이렇게 썼는가
- JD가 "사용자 피드백 기반 UI 개선"을 우대하고, "Claude Code를 활용하는 AI 네이티브 환경"을 강조 → Hotjar 사례와 Claude skills를 구체적으로 언급
- "매주 새로운 기능이 출시되는 환경" → 이력서의 "빠르게 만들어 시장 반응을 확인해야 하는 상황"과 업무 페이스로 연결
- 의료 도메인 경험이 없으므로 도메인 연결은 시도하지 않고, 업무 방식으로 연결

### 핵심 매칭 근거
| JD 요구사항 | 이력서 근거 |
|------------|-----------|
| 인하우스 제품 반복 개선 (2년 이상) | 뭉클랩 1년 10개월, 제품 전 영역 |
| AI 네이티브 (Claude Code 명시) | Claude skills 구축 + 비개발자 인수인계 |
| 사용자 피드백 기반 UI 개선 (우대) | Hotjar 녹화에서 행동 확인 → 성능 개선 |
| 번들러 최적화 (우대) | 10MB → 2MB |
| Zustand 상태관리 (우대) | Skills에 Zustand 기재 |
| 복잡한 문제를 단순하게 풀어내기 | 요금제 13개 페이지 분산 → 설정 일원화 |

### 면접에서 어필할 포인트
1. **사용자 행동을 직접 관찰하고 문제를 발견한 경험** — Hotjar 녹화에서 흰 화면 반복 새로고침 발견 → 성능 개선
2. **데이터 정확성에 대한 감각** — 숫자 0과 미입력 구분 문제 (의료 도메인에서 데이터 정확성은 핵심)
3. **빠른 출시 환경에서 품질과 속도 사이의 판단 경험** — 결제 전환 중 "서비스를 멈추지 않는 쪽을 택하는 판단"
4. **Claude skills로 비개발자 인수인계** — AI를 팀 전체로 확장한 경험

## 3. 스펙터 ★★★★☆

- JD가 "코드 품질을 고민하는 분", "컴포넌트 구조와 상태 흐름을 개선해보려는 분"을 강조 → 품질/일정 판단 경험으로 연결

### 핵심 매칭 근거

## 4. 센트비 ★★★★☆

### 지원 동기
> 이커머스 SaaS에서 결제 시스템 전환, 스타일 토큰화와 공통 컴포넌트 정리, 기능 설계부터 배포까지 1인 프론트엔드로 주도해왔습니다. 기존 방식을 돌아보고 구조를 개선하면서 서비스를 계속 전진시켜온 경험이, 파트너 연동과 신규 기능을 빠르게 만들어야 하는 센트비의 환경과 이어지는 부분이라 지원했습니다.

### 왜 이렇게 썼는가
- JD가 "문제를 스스로 정의하고 주도적으로 해결", "기존 시스템 개선"을 강조 → 결제 전환, 스타일링 전환 등 "기존 방식을 돌아보고 개선한" 경험으로 연결
- 센트비의 기술 스택에 Tailwind, pnpm이 있어 스타일링 전환 경험이 직접 매칭
- "파트너 연동"은 JD 주요업무의 핵심이라 언급했지만, 자기 경험과 억지로 연결하지 않고 "환경"으로만 표현

### 핵심 매칭 근거
| JD 요구사항 | 이력서 근거 |
|------------|-----------|
| 문제를 스스로 정의하고 주도적 해결 | 요금제 구조 문제를 스스로 판단, Hotjar에서 성능 문제 발견 |
| 기존 시스템 개선 경험 (우대) | 요금제 구조, 스타일링, GTM 모두 기존 시스템 개선 |
| 초기 설계부터 출시까지 End-to-End (우대) | 크롬 익스텐션 기획~개발 + 1인 프론트 전 영역 |
| 스타일 토큰화/컴포넌트 구조화 (우대) | MUI→Tailwind 전환, 토큰화, 공통 컴포넌트 정리 |
| Tailwind (기술 스택) | 전환 경험 |

### 면접에서 어필할 포인트
1. **결제 시스템 전환 경험** — 센트비는 해외 송금/결제 서비스이므로 결제 도메인 경험이 직접 도움
2. **기존 방식을 개선한 구체적 사례 3개** — 요금제(구조), 스타일링(체계), GTM(관심사 분리)
3. **운영 중 서비스를 멈추지 않고 전환한 경험** — 결제 PG 전환 시 "차를 두 대 굴린" 접근

- 무촌은 철거/폐기물이라는 "정제되지 않은" 시장을 디지털로 전환 중 → 이력서의 "빠르게 만들어 시장 반응을 확인해야 하는 상황"과 업무 페이스로 연결
- 특수 기술보다 주도성, 전체 사이클, 성능 최적화를 강조하는 JD → 1인 프론트 경험으로 자연스럽게 매칭

### 핵심 매칭 근거
| JD 요구사항 | 이력서 근거 |
|------------|-----------|
| 기획~배포 전체 사이클 (우대) | 크롬 익스텐션 + 1인 프론트 |
| 성능 최적화 (우대) | 10MB → 2MB |
| 주도적 개발 | 1인 프론트엔드 전 영역 |

### 면접에서 어필할 포인트
1. **혼자서 전체 사이클을 돌려본 경험** — 기획~배포~운영까지 빠르게 반복
2. **3년차 스타트업과 비슷한 환경에서 일해봤다** — 빠른 출시, 시장 검증
