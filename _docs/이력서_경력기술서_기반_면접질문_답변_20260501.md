## 1. “더 이상 FE가 병목이 되어선 안 된다”는 표현을 썼는데, 본인이 말하는 FE 병목은 정확히 뭐였나요?

### 답변

제가 말한 FE 병목은 단순히 화면 구현이 느린 문제라기보다, 기능을 시작할 때마다 반복되는 초기 세팅, 페이지마다 흩어진 정책 분기, 스타일 기준 부재, 특정 상태를 다시 만들어야만 가능한 검증처럼 개발과 QA 속도를 계속 늦추는 구조적 비용이었습니다. 실제로 PMF를 찾는 단계에서는 기능 하나를 빨리 만드는 것보다, 가설이 나왔을 때 바로 화면으로 옮겨 팀이 같이 보고 판단할 수 있는 상태를 만드는 게 더 중요하다고 느꼈습니다. 그래서 저는 새 기능을 하나 더 빨리 만드는 것보다, 다음 기능도 빨리 시작하고 검증할 수 있게 만드는 구조를 같이 정리하는 쪽으로 일했습니다.

### 예상 꼬리질문

- 구현 속도와 FE 병목은 뭐가 다른가요?
- 실제로 가장 크게 느낀 병목 하나만 고르면 뭐였나요?
- 왜 지금은 개발 속도보다 의사결정 속도가 더 큰 병목이라고 보나요?

## 2. PMF를 찾는 단계의 1인 프론트엔드로 일했다는 게 실제로는 어떤 역할이었나요?

### 답변

요구사항이 처음부터 깔끔하게 정리된 경우보다, 방향은 있는데 아직 제품 형태로 검증되지 않은 경우가 더 많았습니다. 그래서 저는 주어진 화면만 구현하는 역할보다는, 정제되지 않은 요구사항을 실제 서비스 기능과 검증 가능한 화면으로 바꾸는 역할에 가까웠습니다. 어떤 데이터를 보여줘야 판단이 가능한지, 어떤 상태 분기가 생기는지, 이후 수정 포인트가 어디가 될지를 먼저 보고, 그다음에 빠르게 화면으로 옮기는 식으로 일했습니다. 1인 프론트엔드였기 때문에 구현뿐 아니라 기획 논의, API 연동, QA 환경, 운영 이슈 정리까지 같이 보는 경우가 많았습니다.

### 예상 꼬리질문

- 어디까지 직접 판단했고 어디부터는 의사결정자와 맞췄나요?
- 1인 프론트엔드로 일하면서 가장 어려웠던 점은 뭐였나요?
- 빠르게 만든 화면과 검증 가능한 화면의 차이는 뭐라고 보나요?

## 3. 대표가 엑셀로 보던 가설을 Apollo/GraphQL 기반 4개 리포트로 옮겼다고 했는데, 이 작업의 성격은 뭐였나요?

### 답변

이 작업은 완성도 높은 정식 제품 기능을 깊게 설계했다기보다, 대표님이 엑셀로 보고 있던 매출·광고·비용·상품 가설을 서비스 안에서 반복적으로 검토할 수 있는 형태로 빠르게 옮겨보는 실험성 작업에 가까웠습니다. 저는 그 가설을 UI로 붙이고, 필요한 데이터는 기존 흐름을 그대로 재사용하기보다 Apollo와 GraphQL을 연결해 별도 리포트 흐름으로 가져왔습니다. 중요한 점은 비즈니스 가설 자체를 제가 만든 게 아니라, 그 가설을 검증 가능한 화면과 데이터 흐름으로 빠르게 구현한 역할이었다는 점입니다.

### 예상 꼬리질문

- 왜 기존 REST 흐름을 그대로 쓰지 않고 Apollo/GraphQL을 붙였나요?
- 이걸 제품 기획으로 봐야 하나요, 실행 작업으로 봐야 하나요?
- 1~2일 안에 만들었다는 건 어디까지 구현했다는 뜻인가요?

## 4. 요금제·권한 규칙을 한 곳에서 관리하는 구조로 바꾼 이유는 뭐였나요?

### 답변

기존에는 플랜과 접근 조건이 여러 페이지에 흩어져 있어서 신규 요금제나 정책 변경이 들어오면 수정 포인트를 다시 찾아야 했습니다. 이 방식은 당장은 버틸 수 있어도 정책이 자주 바뀌는 상황에서는 누락 위험과 변경 비용이 계속 커집니다. 그래서 페이지 안의 인라인 분기 대신, 접근 정책은 공통 구조에서 관리하고 화면은 그 결과를 소비하게 바꿨습니다. 사용자에게 드러나는 변화는 크지 않을 수 있지만, 팀 입장에서는 정책 수정 컨텍스트를 한 번에 전달하고 변경 범위를 예측하기 쉬워졌다는 점이 가장 큰 개선이었습니다.

### 예상 꼬리질문

- 기존에는 어떤 문제가 반복됐나요?
- 왜 이걸 라우트와 정책 중심으로 옮겼나요?
- 사용자 화면이 크게 안 바뀌는데 왜 중요한 작업이라고 보나요?

## 5. QA 상태 재현용 mock 데이터 환경으로 복잡한 상태 재현 시간을 최대 99% 줄였다고 했는데, 어떤 문제를 해결한 건가요?

### 답변

결제, 요금제, 쇼핑몰 연동, 광고 연동처럼 상태 조합이 많은 화면은 특정 케이스를 다시 보려면 실제 데이터나 어드민 상태를 매번 맞춰야 했습니다. 그러다 보니 구현보다 검증과 회귀 확인이 더 오래 걸리는 경우가 많았습니다. 그래서 자주 확인해야 하는 조합을 로컬에서 바로 바꿔볼 수 있는 mock 데이터 환경과 분기 상태 재현 도구를 정리했습니다. 핵심은 API를 흉내 내는 것 자체보다, 프론트가 실제로 확인해야 하는 상태 조합을 빠르게 재현해 검증 비용을 줄인 것이었습니다.

### 예상 꼬리질문

- MSW 같은 일반적인 mock 방식과 뭐가 다른가요?
- 99%라는 표현은 어떤 맥락에서 나온 건가요?
- mock 환경이 실제 데이터와 괴리되지는 않았나요?

## 6. Tailwind 기반 스타일 체계와 반복 UI 기준을 정리한 게 왜 병목 해결과 연결되나요?

### 답변

스타일 기준이 정리되지 않으면 화면을 만들 때마다 비슷한 간격, 레이아웃, 토큰 사용을 다시 판단해야 하고, 그 판단 비용이 페이지마다 누적됩니다. 특히 styled나 sx 중심 패턴이 섞여 있으면 사람도 수정 기준이 흔들리고, AI가 작업할 때는 어떤 패턴을 따라야 하는지가 더 모호해집니다. 그래서 신규 코드와 정리 구간부터 Tailwind와 디자인 토큰을 우선 적용하고, 페이지 래퍼, 로딩, 빈 상태 같은 반복 UI를 공통 컴포넌트로 정리했습니다. 핵심은 예쁘게 바꾸는 게 아니라, 이후 화면 작업과 AI 활용이 더 예측 가능해지도록 만든 것입니다.

### 예상 꼬리질문

- 왜 MUI 기반 패턴이 병목이 된다고 봤나요?
- 전면 교체가 아니라 점진 전환을 택한 이유는 뭐였나요?
- Figma MCP 기반 초안 생성과는 어떤 관련이 있나요?

## 7. “AI Native 환경을 만든 작업”이라고 했는데, 구체적으로 무엇을 했나요?

### 답변

저는 AI를 그냥 코드 생성 도구로만 쓰지 않았고, AI가 적은 비용으로 기능 시작, 수정, 검증에 기여할 수 있는 환경을 만드는 데 더 집중했습니다. 대표적으로 페이지 기본 구조를 빠르게 만드는 스캐폴더를 두고, 스타일 기준과 토큰 사용 규칙을 점검하는 스크립트를 추가하고, 트래킹도 공통 구조와 타입이 있는 훅 패턴으로 분리했습니다. 이런 작업들은 겉으로는 생산성 도구처럼 보이지만, 실제로는 AI가 틀리기 쉬운 영역을 줄이고 반복 작업을 구조로 고정해 결과 품질을 안정화하는 역할을 했습니다.

### 예상 꼬리질문

- AI를 많이 쓰면 코드 품질이 흔들리지 않나요?
- 페이지 스캐폴더는 어떤 반복 작업을 줄였나요?
- 왜 트래킹 구조 분리도 AI Native 환경 작업으로 보나요?

## 8. 초기 로딩 병목은 어떻게 발견했고 어떻게 해결했나요?

### 답변

기존에는 lazy loading이 거의 없어서 메인 번들이 10MB 이상으로 커져 있었고, 번들 다운로드 중 보여줄 fallback UI도 없어 사용자가 빈 화면을 앱 오류처럼 받아들일 수 있는 상태였습니다. 실제로 Hotjar 세션에서 사용자가 흰 화면을 버그처럼 인식하고 새로고침을 반복하는 장면을 보면서 문제를 더 명확히 확인했습니다. 그래서 라우트 단위 lazy loading, Suspense fallback, Vite manualChunks를 함께 적용해 페이지와 벤더 청크를 분리했고, 로딩 중에도 사용자가 현재 상태를 이해할 수 있는 화면을 제공하도록 바꿨습니다. 중요한 건 단순히 번들 숫자를 줄인 것보다, 사용자가 멈췄다고 느끼는 경험을 줄인 점입니다.

### 예상 꼬리질문

- lazy loading만으로 충분하지 않았나요?
- 모달과 페이지에서 Suspense fallback을 다르게 둔 이유는 뭔가요?
- 성능 최적화에서 사용자 관찰이 왜 중요했다고 보나요?

## 9. 광고 기능을 분석·캠페인·진단 흐름으로 확장했다고 했는데, 단순 대시보드와 뭐가 달랐나요?

### 답변

숫자만 보여주는 대시보드는 사용자가 현황은 볼 수 있어도 그다음 행동으로 이어가기 어렵다고 생각했습니다. 그래서 광고 기능을 단순 지표 화면이 아니라, 현황을 보고, 문제를 해석하고, 다음 액션으로 이어지는 흐름으로 확장했습니다. 분석, 캠페인, 진단을 실제 서비스 흐름으로 만들고, 그 위에 AI 채팅과 코칭노트 연결까지 붙인 이유도 같은 맥락입니다. 이 작업은 기능 하나를 더 붙였다는 의미보다, 광고 문제를 서비스 안에서 해석하고 다음 행동으로 이어지게 만든 사용자 흐름 설계에 더 가깝습니다.

### 예상 꼬리질문

- 왜 AI 채팅과 코칭노트까지 연결했나요?
- 단순 대시보드와 흐름형 기능의 차이는 뭐라고 보나요?

## 10. 수익성 분석 쪽 DataGrid 경험 개선은 어떤 사용자 문제를 해결한 건가요?

### 답변

수익성이나 주문 데이터 화면은 컬럼이 많고 한 번에 비교해야 할 정보가 많아서, 그냥 표만 보여주면 결국 다시 엑셀로 내려받아 보게 됩니다. 그래서 DataGrid 기반 그룹 분석, 컬럼 접기, 전체화면 같은 기능을 붙여 서비스 안에서 직접 읽고 비교하기 쉬운 형태로 바꿨습니다. 구현 과정에서는 처음에 Dialog 기반 전체화면을 시도했다가 그리드가 이중으로 존재하면서 스크롤, 필터, 정렬 상태가 끊기는 문제를 발견했고, 이후 기존 그리드 인스턴스를 유지한 채 `position: fixed`로 확장하는 방식으로 바꿨습니다. 결과적으로 상태를 잃지 않고 조회 경험을 넓히는 쪽으로 정리한 작업이었습니다.

### 예상 꼬리질문

- 왜 처음엔 Dialog를 선택했나요?
- 왜 `position: fixed`가 더 적절하다고 판단했나요?
- transform이나 Fullscreen API 같은 대안은 왜 쓰지 않았나요?

## 11. FE 밖으로 확장해 API 응답과 데이터 구조까지 봤다고 했는데, 어느 수준까지 관여했나요?

### 답변

화면 구현 속도와 정확도를 높이려면, 프론트에서 실제로 필요한 기준에 맞게 응답 구조와 도메인 모델도 정리되어 있어야 한다고 봤습니다. 그래서 광고 기능처럼 규칙이 많은 영역에서는 필요한 필드, 정렬 기준, 상태값 구조, 저장 흐름까지 같이 맞추는 쪽으로 관여했습니다. 물론 백엔드 전체 설계를 제가 주도했다는 뜻은 아니고, 프론트에서 우회 로직을 늘리는 대신 실제로 필요한 데이터 구조를 같이 정리해 기능 흐름이 덜 흔들리게 만든 경험이라고 설명하는 게 정확합니다.

### 예상 꼬리질문

- 프론트에서 우회하지 않고 응답 구조를 같이 손보려 한 이유는 뭔가요?
- 어디까지가 본인 역할이고 어디부터는 백엔드 역할이었나요?

## 12. 크롬 익스텐션 실험은 어떤 의미가 있었나요?

### 답변

이 작업은 서비스 외부 접점에서 소싱 사용자를 데려오는 채널이 실제로 의미가 있는지 확인해보는 실험이었습니다. 브라우저 안에서 바로 마진 계산으로 이어지는 흐름이면 진입 장벽이 낮을 수 있다고 봐서 기획 논의에 참여하고 개발도 맡았습니다. 결과적으로 다운로드는 확보했지만 결제 전환은 낮았고, 그걸 통해 실제 유료 고객 단계와 소싱 단계 사용자의 간극을 확인할 수 있었습니다. 그래서 성공 사례로 포장하기보다, 다운로드와 전환은 다르다는 걸 배운 가설 검증 사례로 설명하는 게 맞습니다.

### 예상 꼬리질문

- 왜 익스텐션이라는 형태를 택했나요?
- 낮은 결제 전환에서 무엇을 배웠나요?
- 다시 한다면 어떤 지표를 먼저 보겠나요?
