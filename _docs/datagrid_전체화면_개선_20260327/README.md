# DataGrid 전체화면 기능 개선 과정

## 1. 배경

### 왜 시작했나
- 서비스의 DataGrid에 컬럼이 수십 개로 많아서 한눈에 보기 힘듦
- 사용자 해상도가 낮으면 더욱 불편
- 대표님이 다른 서비스에서 그리드 전체보기, 컬럼 열고 접기 기능을 보시고 도입 제안

### 두 가지 기능
1. **컬럼 접기/펼치기** — AG Grid 네이티브 `columnGroupShow` 기능 + 커스텀 아이콘
2. **전체화면 보기** — 그리드를 화면 전체로 확장

---

## 2. 컬럼 접기/펼치기

AG Grid의 built-in 기능을 활용했다.

### 사용 방식
```typescript
// children으로 그룹 정의
{
  headerName: '매출',
  children: [
    {
      field: 'saleAmount',
      headerName: '총 매출',
      columnGroupShow: 'close',  // 그룹 닫혀있을 때 표시 (요약)
    },
    {
      field: 'productSaleAmount',
      headerName: '판매가격',
      columnGroupShow: 'open',   // 그룹 펼쳐졌을 때 표시 (상세)
    },
  ],
}
```

### 변경 이력
- **이전**: 커스텀 `expandState` + `HeaderExpandButton` 컴포넌트로 수동 관리
- **이후** (bb2af18a): AG Grid 네이티브 `columnGroupShow`로 전환, CSS로 아이콘만 커스터마이징

## 3. 전체화면 기능 — 핵심 스토리

### 3-1. 첫 번째 접근: MUI Dialog

**선택 이유**: `fullScreen` prop 하나만 주면 바로 전체화면이 되니까 가장 빠르게 구현할 수 있는 방법이었음

```typescript
// 초기 구현
<Dialog fullScreen open={isFullscreen} onClose={toggleFullscreen}>
  <ToolBar ... />
  <DataGrid ... />  {/* 새로운 그리드 인스턴스 */}
</Dialog>
```

### 3-2. 발견된 문제

Dialog는 **Portal 기반**이라 기존 DOM 트리 밖에 별도로 렌더링되는 구조.
그 안에 AG Grid를 **새로 마운트**할 수밖에 없었다.

**결과적으로 발생한 문제:**
- 같은 데이터를 표시하는 그리드가 **2개** 공존
- 전체화면에서 조작한 스크롤/필터/정렬이 **별개 DOM**에서의 작업
- 전체화면을 끄면 원본 그리드에 **아무것도 반영 안 됨** → UX 단절

```
[일반 화면]                    [전체화면 Dialog]
┌─────────────────┐           ┌─────────────────────────┐
│ DataGrid #1     │           │ DataGrid #2 (새 인스턴스) │
│ - 스크롤 위치 A  │           │ - 스크롤 위치 B          │
│ - 필터 상태 X    │           │ - 필터 상태 Y            │
└─────────────────┘           └─────────────────────────┘
        ↑                              ↑
    원본 (유지)                  별개 DOM (상태 공유 안 됨)
```

### 3-3. 해결: position: fixed

원인이 **"그리드가 2개"**라는 게 명확 → Dialog를 바로 걷어냄

```typescript
// 변경 후 구현
<Box
  sx={{
    width: '100%',
    height: '100%',
    ...(isFullscreen && {
      position: 'fixed',
      top: 0,
      left: 0,
      right: 0,
      bottom: 0,
      zIndex: theme => theme.zIndex.modal,
      bgcolor: 'background.default',
      p: 2,
    }),
  }}
>
  <ToolBar ... />
  <DataGrid ... />  {/* 같은 그리드 인스턴스 그대로 */}
</Box>
```

**fixed를 선택한 이유:**
- DOM 트리에서 요소를 이동시키지 않음 → React 상태/ref 그대로 유지
- 뷰포트 기준으로 `top:0, left:0, right:0, bottom:0` → 화면 전체를 덮음
- 이 두 가지를 동시에 만족하는 가장 단순한 방법

```
[일반 화면]                    [전체화면 (fixed)]
┌─────────────────┐           ┌─────────────────────────────┐
│ DataGrid #1     │    CSS    │ DataGrid #1 (동일 인스턴스)    │
│ - 스크롤 위치 A  │ ───────→ │ - 스크롤 위치 A (그대로)       │
│ - 필터 상태 X    │  토글만   │ - 필터 상태 X (그대로)         │
└─────────────────┘           └─────────────────────────────┘
```

### 3-4. 결과
전체화면을 켜고 꺼도 스크롤·정렬·필터 상태가 그대로 유지됨

## 4. 왜 fixed인가 — 다른 대안과의 비교

### transform: translate
- 위치 이동만 가능, 크기 변경은 별도로 해야 함
- 현재 요소 위치 기준 상대 이동이라 `getBoundingClientRect()`로 위치 계산 필요
- **가장 큰 문제**: transform은 새로운 containing block을 생성
  - 자식 요소의 `position: fixed` 기준점이 뷰포트 → 해당 요소로 바뀜
  - AG Grid 내부의 드롭다운, 툴팁, 컨텍스트 메뉴 등 fixed 요소들이 깨질 수 있음

### position: absolute
- 부모 컨테이너 기준이라 레이아웃에 영향 받음
- 뷰포트 전체를 덮으려면 상위 DOM 구조에 의존하게 됨

### Fullscreen API (el.requestFullscreen())
- 브라우저 네이티브 전체화면 (탭바까지 사라짐)
- 원한 건 "페이지 내 전체화면"이지 "브라우저 전체화면"이 아님

### MUI Dialog (기존)
- Portal 기반 → DOM 트리 밖에 렌더링 → 리마운트 발생
- 이미 실패한 방법

### 결론
| 요구사항 | fixed | translate | absolute | Fullscreen API |
|----------|-------|-----------|----------|----------------|
| DOM 안 건드림 | O | O | O | O |
| 뷰포트 전체 덮기 | O | 계산 필요 | 부모 의존 | O (과함) |
| 내부 fixed 요소 안 깨짐 | O | X | O | O |
| 구현 단순성 | O | X | X | X |

## 6. 이력서용 요약

> DataGrid를 고도화한 계기는 서비스 특성상 그리드에 데이터가 많아 한눈에 보기 힘들고, 해상도가 낮은 환경에서는 더욱 불편했기 때문이다. 대표님이 다른 서비스에서 그리드 전체보기, 컬럼 열고 접기 기능을 보시고 도입을 제안하셔서 바로 구현을 시작했다.
>
> 컬럼 접기는 AG Grid에서 제공하는 네이티브 기능에 커스텀 아이콘을 사용하는 방식으로 진행했고, 전체보기는 처음에 MUI Dialog의 fullScreen 옵션을 사용했다. prop 하나만 주면 바로 전체화면이 되니까 가장 빠르게 구현할 수 있는 방법이었다.
>
> 근데 막상 붙여보니까 Dialog가 Portal 기반이라 기존 DOM 트리 밖에 별도로 렌더링되는 구조였고, 그 안에 AG Grid를 새로 마운트할 수밖에 없었다. 결국 같은 데이터를 보여주는 그리드가 2개가 되어버렸고, 전체화면에서 스크롤하고 필터 걸고 정렬한 게 축소하면 다 날아가는 UX 단절이 생겼다.
>
> 원인이 "그리드가 2개"라는 게 명확했으니까 Dialog를 바로 걷어내고, 기존 그리드 컨테이너에 position: fixed를 적용하는 방식으로 바꿨다. 그리드 인스턴스는 하나 그대로 두고 감싸는 박스의 CSS만 토글하는 거라, 전체화면을 켜고 꺼도 스크롤·정렬·필터 상태가 그대로 유지된다.

## 7. 면접 예상 질문

**Q: Dialog를 왜 처음에 골랐어요?**
A: 제일 빨라서요. fullScreen prop 하나면 바로 전체화면이 되니까.

**Q: 왜 fixed를 골랐어요?**
A: DOM 안 건드리면서 전체화면 되는 방법이 그거였으니까요. 그리드 인스턴스를 하나만 유지하는 게 핵심이었습니다.

**Q: transform은 왜 안 썼어요?**
A: transform은 새로운 containing block을 만들어서 AG Grid 내부의 드롭다운이나 툴팁 같은 fixed 요소들이 깨질 수 있습니다. 그리고 위치 계산도 별도로 해야 해서 fixed보다 복잡해지는데 얻는 이점이 없었습니다.

**Q: Fullscreen API는?**
A: 브라우저 탭바까지 사라지는 진짜 전체화면이라 우리가 원하는 "페이지 내 전체화면"과는 달랐습니다.

**Q: 이중 마운트를 상태 동기화로 해결할 수도 있지 않았나요?**
A: 가능은 하지만 스크롤 위치, 필터, 정렬, 선택 상태를 전부 동기화하는 건 AG Grid의 내부 상태까지 관리해야 해서 오히려 복잡도가 올라갑니다. 근본 원인이 "그리드가 2개"인 거니까 1개로 만드는 게 맞았습니다.
