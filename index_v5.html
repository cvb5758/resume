Introduce.

저는 PMF를 찾는 이커머스 SaaS에서 1인 프론트엔드로 일하며, 정제되지 않은 요구사항을 실제 서비스 기능으로 만들고 검증 가능한 화면으로 바꾸는 일을 반복해왔습니다. 기획 논의부터 개발, 출시, 운영까지 제품이 만들어지는 전 과정을 보며, 단순히 화면을 구현하는 것보다 다음 가설을 더 빠르게 확인할 수 있는 상태를 만드는 데 집중해왔습니다.

요구사항이 늘어날수록 FE 안의 반복 비용도 함께 커졌습니다. 그래서 요금제·권한 규칙, 페이지 기본 구조, 반복 UI, 스타일 시스템, 검증용 테스트 데이터 환경처럼 다음 작업의 시작 비용과 검증 비용을 줄이는 구조를 함께 정리해왔습니다.

광고, 결제, 수익성 분석, 복잡한 데이터 조회, 크롬 익스텐션, 비즈니스 마스터처럼 요구사항의 형태는 매번 달랐지만, 기능 구현과 UI 완성도를 끝까지 맞춰 실제 서비스 안에서 동작하는 기능으로 완성해왔습니다.

저는 어떤 요구사항이 와도 디자인 의도와 기능 요구사항을 함께 이해하고, AI Native한 방식으로 구조화·구현·검증까지 밀고 가는 프론트엔드입니다. 빠르게 만들되 결과물이 흔들리지 않도록 기준을 세우고, 팀이 바로 검증할 수 있는 제품 상태로 만드는 것을 중요하게 생각합니다.

React / Component / State.

  • 페이지는 조립, 컴포넌트는 표현, 훅은 로직과 API 연결을 맡도록 분리합니다. 화면에 필요한 데이터와 핸들러는 커스텀 훅에서 만들고, 컴포넌트는 렌더링과 인터랙션 흐름이 드러나도록 유지합니다.
  • 컴포넌트는 크기보다 역할과 변경 이유를 기준으로 나눕니다. 여러 페이지에서 같은 역할로 반복되는 UI는 공통 레벨로 올리고, 특정 페이지나 도메인 흐름에 묶인 UI는 해당 기능 폴더 안에서 관리합니다.
  • 서버 상태는 React Query로 일관되게 다루되, 대시보드·주문 목록처럼 캐시가 사용자 경험을 높이는 영역과 결제처럼 최신 상태가 중요한 영역을 구분해 캐시 적용 여부를 판단합니다.
  • Zustand 같은 전역 클라이언트 상태는 기본적으로 선호하지 않고, 새 페이지는 전역 상태가 필요하지 않도록 설계합니다. 다만 계정 연동처럼 여러 페이지에서 같은 쇼핑몰 연결 상태를 공유해야 하는 경우에만 제한적으로 사용합니다.

  • 초기 로딩에서 10MB 이상이던 메인 번들 때문에 화면이 비어 보이고 사용자가 앱이 멈췄다고 느껴 새로고침을 반복하는 문제가 있었습니다. 이를 줄이기 위해 라우트 단위 lazy loading, Suspense fallback, Vite manualChunks로 페이지·벤더 청크를 분리했습니다.
