Introduce.

저는 PMF를 찾는 이커머스 SaaS에서 1인 프론트엔드로 일하며, 정제되지 않은 요구사항을 실제 서비스 기능으로 만들고 검증 가능한 화면으로 바꾸는 일을 반복해왔습니다. 최근에는 수준 높은 구현 자체를 AI가 빠르게 해내는 경우가 많아졌고, 과거에는 아이디어를 실제 제품으로 옮기는 개발 자체의 속도가 병목이었다면, 지금은 무엇을 만들지 정하고 빠르게 검증하는 의사결정의 속도가 더 중요한 병목이 되어가고 있다고 느끼고 있습니다.

저는 단순히 화면을 빠르게 붙이는 FE라고 생각하지 않습니다. AI로 구현 속도를 높일 수는 있지만, 구조와 설계가 정리되어 있지 않으면 결과물은 쉽게 흔들린다고 생각합니다. 그래서 반복 가능한 구조와 검증 가능한 상태를 함께 만드는 데 더 집중해왔고, 이전부터 서비스의 성장을 위해 어떤 기능이 실제로 제품 지표와 운영 흐름에 영향을 줄지 함께 고민해왔습니다.

그 과정에서 지금 제가 가장 직접적으로 해결할 수 있는 병목은, FE 기본 구현 레벨의 생산성을 극단적으로 줄여 가설이 나오면 바로 화면으로 옮겨 이야기할 수 있는 상태를 만드는 일이라고 판단했습니다. 그래서 AI 레버리지를 높일 수 있는 코드 구조, 반복 작업 자동화, Tailwind 기반 스타일 체계 정리, QA 상태 재현용 mock 데이터 환경처럼 다음 작업의 시작 비용과 검증 비용을 함께 낮추는 구조를 계속 정리해왔습니다.

지금은 어떠한 가설이 나오더라도 하루 안에 직접 화면으로 보며 이야기할 수 있는 수준까지 FE 병목을 줄이는 방향으로 일하고 있습니다. 더 이상 FE가 병목이 되어선 안 된다고 생각합니다. FE의 병목을 줄이고, 팀이 제품 자체에 더 오래 집중할 수 있게 만들고 싶은 Frontend Engineer 이수민입니다.

React.

  • 페이지는 조립, 컴포넌트는 표현, 훅은 로직과 API 연결을 맡도록 분리함. 화면에 필요한 데이터와 핸들러는 커스텀 훅에서 만들고, 컴포넌트는 렌더링과 인터랙션 흐름이 드러나게 유지함.
  • 컴포넌트는 크기보다 역할과 변경 이유를 기준으로 나눔. 여러 페이지에서 같은 역할로 반복되는 UI는 공통 레벨로 올리고, 특정 페이지나 도메인 흐름에 묶인 UI는 해당 기능 폴더 안에서 관리함.
  • 서버 상태는 React Query로 일관되게 다룸. 대시보드·주문 목록처럼 캐시가 사용자 경험을 높이는 영역과 결제처럼 최신 상태가 중요한 영역을 구분해 캐시 적용 여부를 판단함.
  • Zustand 같은 전역 클라이언트 상태는 기본적으로 선호하지 않음. 새 페이지는 전역 상태가 필요하지 않도록 설계하고, 계정 연동처럼 여러 페이지에서 같은 쇼핑몰 연결 상태를 공유해야 하는 경우에만 제한적으로 사용함.

Styling.

  • 레이아웃은 기본적으로 flex를 중심으로 잡음. 각 요소가 다른 레이아웃과 강하게 엮이지 않도록 스타일 책임 범위를 좁혀 구현함.
  • 특정 요소의 스타일을 수정했을 때 주변 레이아웃에 예상치 못한 변화가 생기지 않도록, 간격·정렬·크기 기준을 명확히 두고 CSS 수정이 쉬운 구조를 중요하게 봄.
  • 반복되는 스타일 판단은 토큰과 공통 컴포넌트로 올림. 한 화면에만 필요한 UI는 도메인 맥락이 유지되도록 과하게 일반화하지 않음.

Performance.

  • 초기 로딩에서 10MB 이상이던 메인 번들 때문에 화면이 비어 보이고 사용자가 앱이 멈췄다고 느껴 새로고침을 반복하는 문제가 있었음. 이를 줄이기 위해 라우트 단위 lazy loading, Suspense fallback, Vite manualChunks로 페이지·벤더 청크를 분리함.
  • 기존에는 데이터 로딩 시 페이지 전체가 한 번에 로딩 상태로 전환되어 사용자가 필요한 영역까지 기다려야 했음. 이를 컴포넌트 단위 Suspense fallback으로 나눠 먼저 보일 수 있는 영역은 바로 노출하고, 늦게 오는 영역만 부분적으로 로딩되도록 바꿔 체감 로딩 UX를 개선함.
